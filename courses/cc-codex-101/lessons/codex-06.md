## 📚 학습 목표 — 4가지 도달점

1. **한 번 써두면 매번 설명하지 않아도 된다** — AGENTS.md 규칙서가 생기면 "저는 비개발자입니다, 한국어로 써주세요"를 새 세션마다 반복하지 않아도 Codex가 처음부터 지우의 방식대로 일한다.
2. **CLAUDE.md와 AGENTS.md는 이름만 다른 쌍둥이** — cc-05에서 배운 것의 90%가 그대로 전이된다는 것을 직접 손으로 확인한다.
3. **규칙이 있으면 답이 달라진다** — AGENTS.md 없이 시킨 것과, 만들고 나서 시킨 것의 응답 차이를 나란히 눈으로 본다.
4. **config.toml로 기본값을 고정한다** — 매번 모델과 effort를 손으로 고르지 않고, 파일 한 장에 저장해 두면 다음 세션부터 자동 적용된다.

## 🎯 개념 도입 — 새 알바생에게 주는 업무 매뉴얼

### AGENTS.md가 없으면 어떤 일이 벌어지나

지우는 Codex를 켤 때마다 이런 말을 앞에 붙이고 있었습니다.

> "저는 온라인 강의랑 뉴스레터를 하는 1인 사업자고요, 비개발자라 기술 용어는 쉽게 설명해 주시고, 보고는 한국어로 해주세요. 파일 지우기 전에는 꼭 먼저 물어봐 주세요. 분석 끝에는 다음에 뭘 하면 좋을지 제안도 한 줄 넣어 주세요…"

매번, 똑같이. 새 세션을 열 때마다.

이건 마치 새 알바생이 가게에 들어올 때마다 "여기서는 반드시 영수증을 드리고, 카드 결제는 이쪽 단말기로, 고객 이름은 꼭 물어봐야 해요"를 처음부터 다시 설명하는 것과 같습니다.

한 번 **업무 매뉴얼을 써서 건네주면** 어떻게 될까요? 그 알바생은 매번 같은 규칙을 묻지 않고, 처음부터 그 방식대로 일합니다.

**AGENTS.md가 바로 그 업무 매뉴얼**입니다.

### CLAUDE.md와 AGENTS.md — 이름만 다른 같은 것

cc-05에서 `CLAUDE.md`를 배웠습니다. "Claude Code가 내 프로젝트에서 지켜야 할 규칙을 적는 파일"이었죠.

`AGENTS.md`는 Codex 버전의 그 파일입니다. 역할, 규칙, 보고 형식, 금지사항 — 적는 내용도 같고, "이 폴더에 이 파일이 있으면 Codex가 자동으로 읽는다"는 작동 방식도 같습니다. 차이는 딱 하나, 파일 이름입니다.

| 항목 | Claude Code | Codex |
|---|---|---|
| 규칙서 이름 | `CLAUDE.md` | **`AGENTS.md`** |
| 역할 | CC가 내 방식대로 일하게 하는 매뉴얼 | Codex가 내 방식대로 일하게 하는 매뉴얼 |
| 담는 내용 | 내 사업 맥락·말투·금지사항·완료기준 | 똑같음 |
| 어디에 두나 | 프로젝트 폴더 루트 | 프로젝트 폴더 루트 또는 `~/.codex/` |
| 뼈대 자동 생성 | (직접 작성 또는 첫 실행 시) | 세션 안에서 `/init` 입력 |

**결론**: CLAUDE.md를 한 번 만들어 봤다면, AGENTS.md는 "파일 이름만 바꿔서 다시 써보는" 실습입니다. 개념이 전이되는 것을 손으로 확인하는 게 이번 레슨의 핵심입니다.

### 우선순위 — "이 프로젝트 규칙이 기본 규칙보다 강하다"

AGENTS.md는 여러 위치에 둘 수 있습니다. **가장 가까운(프로젝트에 특화된) 것이 이깁니다.**

```
AGENTS.override.md  >  ./AGENTS.md  >  ../AGENTS.md  >  ~/.codex/AGENTS.md
(강제 최우선)          (지금 폴더)      (상위 폴더)      (내 컴퓨터 전체 기본)
```

지금 단계에서 기억할 것은 하나입니다: **"작업 폴더에 AGENTS.md 하나 두면 된다."** 나중에 여러 프로젝트를 관리할 때 이 우선순위가 유용해집니다.

### config.toml — 단골 식당의 저장된 주문

`config.toml`은 Codex의 기본 설정 파일입니다.

비유: 자주 가는 순댓국집에서 사장님이 "덜 맵게, 양 많이"를 기억해두는 것. 들어서기만 해도 그 주문이 나옵니다.

매번 터미널에 `-m gpt-5.5 --ask-for-approval on-request`를 길게 치는 대신, `config.toml`에 한 번 적어두면 다음부터 `codex --cd ~/강의관리`만 쳐도 그 설정이 자동 적용됩니다.

```toml
# ~/.codex/config.toml
model = "gpt-5.5"
model_reasoning_effort = "medium"
approval_policy = "on-request"
```

읽는 법 한 줄씩:
- `model = "gpt-5.5"` → 기본 모델은 gpt-5.5 (균형 잡힌 기본값)
- `model_reasoning_effort = "medium"` → 기본 생각 강도는 중간 (대부분 작업에 적당)
- `approval_policy = "on-request"` → 모델이 필요하다고 판단할 때 확인 요청 (안전)

## 🔧 실습 1 — AGENTS.md 만들고 차이 직접 확인하기

**이 실습의 목표**: 규칙서가 있을 때와 없을 때 Codex의 답이 어떻게 달라지는지 눈으로 확인한다. 실습 1은 처음부터 끝까지 **대화형(세션) 방식**으로 진행합니다 — 매 단계마다 Codex를 대화형으로 실행한 뒤 그 안에서 질문을 입력합니다.

### 1단계: 규칙서 없이 먼저 시켜보기 (비교 기준 만들기)

**왜 이걸 먼저 하나**: 규칙서가 있을 때의 차이를 몸으로 느끼려면, 없을 때의 답이 어떤지 먼저 봐야 합니다.

터미널을 열고 아래처럼 Codex 대화형 세션을 시작하세요. `--cd` 뒤에 여러분의 실제 폴더 경로를 적으면 됩니다.

```bash
codex --cd ~/[여러분의-경로]/practice-data/creator
```

Codex가 켜지면, 아래 질문을 입력하세요.

```text
student_inquiries.csv를 읽고, 이번 달 문의에서 가장 많이 들어온 문의 유형 3가지를 정리해 줘
```

**기대 결과 예시** (AGENTS.md 없을 때):
```
student_inquiries.csv를 확인했습니다.

Top 3 inquiry types in March 2026:
1. Practice/Setup issues (터미널, AGENTS.md 위치, WSL 오류)
2. Payment issues (결제 확인, 환불, 할인코드)
3. Course access issues (강의 안 열림, 목록 미표시)
```

**관찰 포인트**: 답이 왔지만 영어가 섞이거나, 지우의 사업 맥락이 없거나, 제안이 없을 수 있습니다. 이 답을 기억해두세요(스크린샷을 찍어도 좋습니다). 잠시 뒤 AGENTS.md를 만들고 같은 질문을 다시 던질 겁니다.

Codex 세션을 종료(`Ctrl+C` 또는 `/exit`)하고 다음 단계로 갑니다.

### 2단계: AGENTS.md 뼈대 자동 생성 (세션 안에서 `/init`)

creator 폴더에서 Codex 대화형 세션을 다시 시작하세요.

```bash
codex --cd ~/[여러분의-경로]/practice-data/creator
```

Codex가 켜지면 (터미널이 아닌 **Codex 세션 안의 입력창**에서) 입력하세요.

```text
/init
```

**왜 이걸 하나**: `/init`은 Codex 세션 안에서 입력하는 슬래시 명령입니다 — 터미널 셸 명령이 아닙니다. 입력하면 Codex가 "지금 폴더를 둘러보고 AGENTS.md의 기본 틀을 자동으로 만들어줘"라고 이해합니다. 처음부터 직접 쓰는 것보다 훨씬 쉽습니다.

**기대 결과**: Codex가 폴더를 잠깐 살펴본 뒤 AGENTS.md 초안을 만들어 줍니다. 대략 이런 내용이 들어갑니다.

```markdown
# AGENTS.md (자동 생성 초안)

## Project Overview
이 폴더에는 강의별 매출 데이터, 뉴스레터 통계, 수강생 문의 내역과
콘텐츠 기획 문서, 리서치 메모, 일정 노트가 있습니다.

## Files
- course_sales_monthly.csv
- newsletter_stats.csv
- student_inquiries.csv
- content_plan.md / research_memo.md / consulting_schedule.md

## Rules
...
```

**관찰 포인트**: Codex가 폴더 안을 실제로 읽고 파일을 파악했음을 알 수 있습니다. 하지만 이건 자동 초안입니다 — 지우만의 규칙을 직접 추가해야 합니다.

### 3단계: AGENTS.md에 지우의 규칙 추가하기

텍스트 편집기로 `practice-data/creator/AGENTS.md` 파일을 열고, 아래 내용을 통째로 붙여넣습니다. (기존 자동 생성 내용을 대체해도 됩니다.)

```markdown
# AGENTS.md — 지우의 콘텐츠 사업 운영 규칙서

## 내가 누구인가
나는 온라인 강의·뉴스레터·컨설팅을 운영하는 1인 사업자 지우다.
직원이 없고, 기획·글쓰기·고객응대·매출정리·리서치를 혼자 다 한다.
코딩은 모른다. 비개발자다.

## Codex의 역할
- 강의별 매출 데이터(course_sales_monthly.csv) 분석
- 수강생 문의(student_inquiries.csv) 패턴 정리
- 뉴스레터 성과(newsletter_stats.csv) 분석
- 콘텐츠 기획 문서(content_plan.md) 검토 및 제안
- 분석 결과를 바탕으로 보고서 초안 작성

## 반드시 지킬 규칙
- 모든 답변은 비개발자가 읽기 쉬운 한국어로.
- 수강생 이름, 연락처 등 개인정보는 절대 외부에 노출하지 말 것.
- 파일을 삭제하거나 덮어쓰기 전에는 반드시 먼저 물어볼 것.
- 어려운 용어가 나오면 비유로 먼저 설명할 것.

## 보고서 형식
분석 결과는 항상 이 순서로:
1. 관찰 — 데이터에서 발견한 것
2. 해석 — 그게 왜 일어났는지
3. 제안 — 그래서 다음에 무엇을 하면 좋은지

## 완료 기준
- 보고서에는 반드시 "그래서 다음에 뭘 하면 되는지" 한 줄이 포함돼야 한다.
- 숫자를 언급할 때는 실제 파일의 수치를 인용할 것 (지어내지 말 것).
- 핵심 3가지 + 제안 1줄이 긴 답변보다 낫다.
```

저장 후 Codex를 새로 시작합니다.

```bash
codex --cd ~/[여러분의-경로]/practice-data/creator
```

**왜 재시작하나**: Codex는 세션을 시작할 때 AGENTS.md를 읽습니다. 파일을 고친 뒤 재시작해야 바뀐 규칙이 반영됩니다.

내가 누구야?

### 5단계: 같은 질문 다시 던져보기 (차이 확인)

이제 1단계와 **완전히 똑같은** 질문을 다시 합니다.

```text
student_inquiries.csv를 읽고, 이번 달 문의에서 가장 많이 들어온 문의 유형 3가지를 정리해 줘
```

**기대 결과 예시** (AGENTS.md 적용 후):
```
student_inquiries.csv를 확인했습니다.
2026년 3월 총 24건 문의 중 가장 많은 유형 3가지입니다.

**관찰**
1. 실습·설치 관련 (8건, 33%) — 배준혁 WSL 3.0시간, 정유나 터미널 에러 2.0시간 외
   주로 "클로드코드로 업무 자동화 입문" 수강생. AGENTS.md 위치, WSL 오류, Codex 로그인 반복.

2. 결제·환불 관련 (7건, 29%) — 김하늘, 이도윤, 오현우 외
   결제 미열람, 중복결제, 할인코드 오류, 환불 요청. 우선순위 높음 3건 포함.

3. 강의 문의 (5건, 21%) — 최민서, 서다은, 문가영 외
   수강 전 질문. "비개발자도 가능한가요?", 수강 기간, 환불 규정, 단체 할인 등.

**해석**
실습·설치 문의가 1/3을 차지하며, 배준혁(WSL 3시간)·정유나(터미널 2시간)·백서윤(Codex 로그인 2시간) 처리에 가장 많은 시간이 들었습니다. 새 강의 런칭 직후라 설치 관련 혼선이 집중됐습니다.

**제안**
터미널 설치 FAQ(Mac용·Windows WSL용 분리)를 강의 시작 전 안내 페이지로
만들면 이 유형 문의의 절반을 줄일 수 있습니다.
```

**핵심 관찰 포인트**: 1단계 답과 비교해 보세요.
- AGENTS.md가 없을 때: 영어 섞인 목록, 제안 없음, 수치 없음.
- AGENTS.md가 있을 때: 한국어, 관찰→해석→제안 형식, 실제 수치(33%, 29%, 21%) 인용, 다음 행동 제안.

같은 질문인데 답이 달라졌습니다. 이게 규칙서의 힘입니다.

### 6단계: "완료 기준" 규칙이 실제로 강제되는지 체험하기

### 7단계: AGENTS.md를 두 위치에 두고 우선순위 직접 체험하기

앞서 설명한 우선순위 — "가까운 폴더의 AGENTS.md가 이긴다" — 를 실제로 손으로 체험합니다.

먼저, `~/.codex/` 폴더에 전체 기본 AGENTS.md를 하나 만들어 봅니다.

```bash
mkdir -p ~/.codex
```

텍스트 편집기로 `~/.codex/AGENTS.md` 파일을 새로 만들고 아래를 입력하세요.

```markdown
# AGENTS.md — 내 컴퓨터 전체 기본 규칙

## 기본 규칙
- 모든 답변은 한국어로.
- 파일 수정 전에는 반드시 물어볼 것.
- 보고서 형식: 요약만 (관찰·해석·제안 없음).
```

저장 후 Codex를 새로 시작합니다. 이번엔 creator 폴더에서 실행합니다.

```bash
codex --cd ~/[여러분의-경로]/practice-data/creator
```

그리고 다시 물어봅니다.

```text
내가 누구야? 그리고 보고서는 어떤 형식으로 써야 해?
```

**기대 결과 — 프로젝트 AGENTS.md가 이겼을 때**:
```
지우입니다. 1인 사업자로 강의·뉴스레터·컨설팅을 운영하십니다.

보고서 형식은 "관찰 → 해석 → 제안" 순서입니다.
```

**관찰 포인트**: 전체 기본 규칙(`~/.codex/AGENTS.md`)에는 "요약만"이라고 적었는데, 실제로는 프로젝트 폴더의 `AGENTS.md`("관찰→해석→제안")가 적용됐나요? 가까운 폴더의 규칙이 이겼습니다.

이것이 우선순위입니다: **프로젝트별 맞춤 규칙이 컴퓨터 전체 기본 규칙보다 강합니다.**

실습이 끝나면 `~/.codex/AGENTS.md`는 삭제하거나 내용을 비워도 됩니다 — 지금은 우선순위 체험이 목적이었습니다.

## 📝 실습 3 — config.toml로 기본값 고정하기 (선택·시간 남으면)

AGENTS.md가 "어떻게 일할지"라면, `config.toml`은 "어떤 도구로 일할지"입니다.

### 12단계: config.toml 만들기

`~/.codex/` 폴더에 `config.toml` 파일을 새로 만들고 아래 내용을 붙여넣습니다.

Mac:
```bash
mkdir -p ~/.codex
```

Windows(WSL):
```bash
mkdir -p ~/.codex
# 파일 탐색기에서 열려면:
explorer.exe $(wslpath -w ~/.codex)
```

텍스트 편집기로 `~/.codex/config.toml` 파일을 새로 만들고 아래 내용을 붙여넣으세요.

```toml
# config.toml — 지우의 Codex 기본 설정

# 기본 모델: gpt-5.5 (균형 잡힌 기본값)
model = "gpt-5.5"

# 기본 생각 강도: medium (대부분의 사업 분석 작업에 적당)
model_reasoning_effort = "medium"

# 승인 정책: 모델이 필요하다고 판단할 때 확인 요청 (on-request = 모델 판단 시 확인)
approval_policy = "on-request"
```

저장 후 Codex를 새로 시작하세요.

```bash
codex --cd ~/[여러분의-경로]/practice-data/creator
```

Codex가 켜지면:

```text
/status
```

**기대 결과**:
```
현재 모델: gpt-5.5
Effort: medium
승인 정책: on-request
현재 폴더: .../practice-data/creator
```

**관찰 포인트**: config.toml에 적어둔 대로 모델과 effort가 적용됐나요? 이제 `codex --cd ~/[폴더]`만 쳐도 이 기본값이 자동 적용됩니다.

### 13단계: 사업 상황에 맞게 설정 조절하기

config.toml의 값은 언제든 바꿀 수 있습니다.

**단순 반복 작업(파일명 정리, 짧은 번역)을 시킬 때**:
```toml
model = "gpt-5.4-mini"
model_reasoning_effort = "low"
```

**중요한 강의 커리큘럼 분석이나 복잡한 보고서를 만들 때**:
```toml
model = "gpt-5.5"
model_reasoning_effort = "high"
```

**지우의 실제 추천 기본값 (일상 업무용)**:
```toml
model = "gpt-5.5"
model_reasoning_effort = "medium"
approval_policy = "on-request"
```

> 💡 **비유**: 비서에게 "오늘 급한 서류 정리는 빠르게, 중요한 제안서는 꼼꼼하게"라고 말하는 것과 같습니다. config.toml은 그 기본 마음가짐을 파일로 적어둔 것입니다.

## 🔍 흔한 오류와 해결법

| 상황 | 원인 | 해결 |
|---|---|---|
| AGENTS.md를 만들었는데 Codex가 무시하는 것 같다 | 파일 위치가 잘못됐거나 실행 폴더가 다름 | 파일이 `practice-data/creator/AGENTS.md`에 있는지 확인. Codex를 `--cd practice-data/creator`로 실행 |
| 파일 이름을 `agents.md`(소문자)로 만들었다 | 대소문자 구분 문제 | 파일 이름은 반드시 `AGENTS.md` (대문자) |
| AGENTS.md를 바꿨는데 반영이 안 된다 | 기존 세션은 이미 읽은 규칙서를 유지 | Codex 세션을 새로 시작(터미널 새로 열기)하면 바뀐 AGENTS.md를 다시 읽음 |
| 세션 안에서 `/init`을 입력했는데 아무것도 안 생긴다 | Codex 버전 차이 또는 명령 입력 위치 오류 | `/init`은 Codex 세션 안 입력창에서 입력해야 합니다 (터미널 셸 ❌). 그래도 안 되면 직접 텍스트 편집기로 AGENTS.md 파일을 만들어도 됩니다 |
| config.toml이 적용 안 된다 | 파일 위치 오류 또는 파일 형식 오류 | `~/.codex/config.toml` 경로를 다시 확인. TOML 형식에서 따옴표·등호가 올바른지 확인 |
| `gpt-5.5` 모델이 목록에 없다 | ChatGPT 구독 등급 문제 또는 버전 차이 | ChatGPT Plus/Pro 구독 확인. `/model`로 사용 가능한 목록을 확인하세요 |
| Codex가 파일 저장 전에 전혀 확인을 안 한다 | config.toml의 `approval_policy` 누락 또는 `never` 설정 | config.toml에 `approval_policy = "on-request"` 추가 후 재시작. on-request는 모델이 판단할 때 확인하므로 모든 저장 전 확인이 보장되지는 않음 |
| "관찰 → 해석 → 제안" 형식이 안 나온다 | AGENTS.md의 "보고서 형식" 섹션이 빠졌거나 세션 재시작 안 함 | AGENTS.md 내용 확인 후 Codex 재시작 |
| `~/.codex/` 폴더를 Mac에서 열고 싶다 | Finder는 `.`으로 시작하는 숨김 폴더를 기본적으로 안 보여줌 | 터미널에서 `open ~/.codex/` 입력. Windows WSL이면 `explorer.exe $(wslpath -w ~/.codex)` |
| 우선순위 체험 단계에서 프로젝트 AGENTS.md가 안 이긴다 | 두 파일 이름이나 경로가 잘못됨 | 프로젝트 폴더의 AGENTS.md는 `practice-data/creator/AGENTS.md`, 기본값 파일은 `~/.codex/AGENTS.md` 인지 확인 |

## 💡 핵심 규칙·개념 정리

### AGENTS.md 담는 내용

| 섹션 | 예시 (지우의 경우) |
|---|---|
| 내가 누구인가 | "온라인 강의·뉴스레터 운영 1인 사업자, 코딩 모름" |
| Codex의 역할 | "매출·문의·뉴스레터 데이터 분석, 보고서 초안 작성" |
| 반드시 지킬 규칙 | "한국어 전용, 실명 노출 금지, 파일 삭제 전 확인" |
| 보고서 형식 | "관찰 → 해석 → 제안 순서" |
| 완료 기준 | "실제 수치 인용, 다음 행동 1줄 포함" |

### config.toml 핵심 키 3개

```toml
model = "gpt-5.5"                  # 기본 모델 (gpt-5.5 권장)
model_reasoning_effort = "medium"   # low/medium/high/xhigh
approval_policy = "on-request"      # 모델 판단 시 확인 요청 (매번 보장 아님)
```

### 우선순위 (가장 가까운 것이 이긴다)

```
AGENTS.override.md > ./AGENTS.md > ../AGENTS.md > ~/.codex/AGENTS.md
config: 프로젝트 .codex/config.toml > ~/.codex/config.toml
```

### CLAUDE.md ↔ AGENTS.md 한눈 대응

| | Claude Code | Codex |
|---|---|---|
| 규칙서 | `CLAUDE.md` | `AGENTS.md` |
| 뼈대 생성 | (직접 작성 또는 첫 실행 시) | 세션 안에서 `/init` 입력 |
| 두는 위치 | 프로젝트 폴더 루트 | 프로젝트 폴더 루트 또는 `~/.codex/` |
| 적는 내용 | 역할·규칙·보고 형식·금지사항 | 동일 |
| 우선순위 원리 | 가까운 폴더가 우선 | 가까운 폴더가 우선 (동일) |

**한 문장 요약**: AGENTS.md는 Codex에게 주는 업무 매뉴얼이고, config.toml은 자주 가는 식당에 저장해 둔 단골 주문이다. CLAUDE.md를 만들어 봤다면 이미 90%는 안다.
