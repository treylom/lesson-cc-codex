# cc-07: GitHub로 저장·공유 — "내 사업 기록이 사라지지 않는 가장 쉬운 방법"

> **레슨 목표**: GitHub를 처음 들어보는 사람도 Claude Code에 "저장해줘", "올려줘" 두 마디로 내 작업물을 안전하게 보관하고, 어느 기기에서든 꺼낼 수 있게 만든다. "코드 관리 도구"가 아니라 **"1인 사업자의 작업 타임캡슐"**로 GitHub를 이해한다.
> **예상 시간**: 50분 (개념 도입 8분 + 실습 1 준비 15분 + 실습 2 분석·저장 18분 + 실습 3 선택 9분)
> **사전 준비**: cc-01~06 완료 상태 / 터미널 열기 / 인터넷 연결 / GitHub 계정 (없으면 github.com 무료 가입 5분)
> **진입 커맨드**: `/cc-07`

---

## 📚 학습 목표 — 4가지 와우 모먼트

1. **"또 날아갔다"가 사라진다** — Claude Code가 만들어 준 보고서·기획 노트·CLAUDE.md 규칙서를 GitHub에 올리면, 컴퓨터가 망가져도 내용이 남는다는 사실을 직접 체험한다.
2. **"저장해줘"·"올려줘" 두 마디면 끝이다** — Git 명령어를 외울 필요 없다. Claude Code에 한국어로 부탁하면 AI가 대신 commit하고 push해 준다는 것을 확인한다.
3. **버전 기록이 생긴다** — "어? 지난주 분석 파일 어디 갔지?"가 생길 때, GitHub 히스토리에서 예전 파일을 꺼내올 수 있다는 것을 체감한다.
4. **어느 기기에서든 내 작업이 보인다** — 카페 노트북, 집 데스크톱, 어느 기기에서도 GitHub 링크 하나면 내 파일이 열린다.

## 🎯 개념 도입

### GitHub가 뭔가요? — 작업 타임캡슐

GitHub를 처음 들으면 "개발자가 코드 올리는 곳"이라는 인상이 있습니다. 맞습니다, 그런 용도도 있습니다. 하지만 지우에게 GitHub가 필요한 이유는 딱 하나입니다.

**비유**: 중요한 서류를 서랍에만 보관하는 대신, **사진도 찍고 외부 창고에도 복사본을 넣어두는 것.**

지우는 cc-03~06에서 Claude Code와 함께 여러 파일을 만들었습니다.
- `course_sales_monthly.csv` 분석 결과 파일
- `CLAUDE.md` 규칙서 (지우의 사업 성격·톤·우선순위가 담긴 핵심 파일)
- 보고서 초안들

이 파일들이 지금 어디 있나요? 지우 컴퓨터 안에만 있습니다. 컴퓨터가 고장나면? 실수로 파일을 삭제하면? → 사라집니다.

GitHub는 이 파일들을 **인터넷 창고(클라우드)**에 올려두는 시스템입니다. 그리고 단순 백업이 아니라, **언제 무엇을 바꿨는지 역사(히스토리)**도 남깁니다.

### Git vs GitHub — 두 이름이 헷갈린다면

처음에 두 이름이 헷갈리는 게 당연합니다. 비유로 먼저 풀겠습니다.

**Git** = 내 컴퓨터 안에서 파일의 변경 이력을 기록하는 **사진 앨범**입니다.
- 파일을 저장할 때마다 "오전 10시 버전", "오후 2시 버전" 식으로 스냅샷(그 순간의 상태)을 찍어 둡니다.
- 실수로 파일을 지워도 앨범을 뒤져서 "어제 오후 버전으로 되돌려줘"가 가능합니다.

**GitHub** = 그 앨범을 **인터넷 클라우드 서버에 올려두는 곳**입니다.
- Google Drive가 내 문서를 클라우드에 저장하는 것처럼, GitHub는 Git 앨범을 클라우드에 올려 둡니다.
- 다른 컴퓨터에서도 내려받을 수 있고, 링크를 공유할 수 있습니다.

### git 도구 vs gh 도구 — 왜 두 가지를 설치하나요?

실습 중에 두 가지 별개 도구가 등장합니다. 미리 구분해두면 헷갈리지 않습니다.

| 도구 | 역할 | 비유 |
|------|------|------|
| **git** | 내 컴퓨터 안에서 파일 이력을 관리하는 앨범 도구 | 사진을 찍고 앨범에 붙이는 행위 자체 |
| **gh** (GitHub CLI) | GitHub 서버(인터넷 창고)에 접속하는 전용 리모컨 | 택배 기사를 부르는 전화기 |

git만 있으면 내 컴퓨터 안에서 버전을 기록할 수 있습니다. 하지만 그것을 GitHub 서버에 **올리고 관리하는 작업**은 gh라는 별도 도구가 훨씬 편리하게 처리합니다. 둘 다 필요합니다. Claude Code가 대신 써 주니까 직접 명령어를 외울 필요는 없지만, 왜 두 가지인지 이해하고 있으면 오류가 나도 당황하지 않습니다.

> 💡 **핵심 안심**: cc-08에서 배우는 플러그인·스킬 중에 'git-teacher'(바르다 깃선생)라는 도구가 있습니다. 그 도구를 쓰면 "깃 시작해줘" 한 마디로 설치·계정 연결·프로젝트 초기화를 자동으로 해 줍니다. 이번 레슨(cc-07)은 플러그인 없이 **Claude Code 대화만으로 하는 방식**이라 단계가 조금 더 있지만, 덕분에 "어떤 일이 일어나는지" 한 번 눈으로 볼 수 있습니다.

### Google Drive vs GitHub — 핵심 차이 한눈에

| 항목 | Google Drive / iCloud | GitHub (Git) |
|------|----------------------|--------------|
| 저장 방식 | **자동** — 타이핑하면 실시간 저장 | **수동** — 내가 "저장" 신호를 보낼 때만 |
| 버전 기록 | 최근 수십 개, 보기 어려움 | 내가 저장할 때마다 **이유를 적어** 기록 |
| "왜 바꿨나" 메모 | 없음 | Commit 메시지(라벨)에 내가 직접 씀 |
| 되돌리기 | 제한적 (날짜 기준) | 아무 버전으로나 정확히 되돌리기 |
| 비용 | 무료 15GB → 유료 | **무료** (비공개 저장소도 무료) |

비개발자인 지우에게 Git이 필요한 이유는 **"AI와 함께 만든 보고서·초안·규칙서가 덮어써지거나 사라져도 언제든 복구할 수 있기 때문"**입니다. 특히 `CLAUDE.md`처럼 오래 다듬어온 규칙서는 실수로 지워지면 처음부터 다시 만들어야 하는데, Git이 연결돼 있으면 명령 한 줄로 돌아옵니다.

### 핵심 용어 3가지 — 이것만 알면 됩니다

| 용어 | 쉬운 말 | 비유 |
|------|---------|------|
| **Repository (저장소)** | GitHub에 만든 내 전용 창고 | Google Drive의 공유 폴더 |
| **Commit (커밋)** | 저장하기 + 메모 달기 | 문서를 포장해서 "오늘 분석 완료" 라벨 붙이기 |
| **Push (푸시)** | GitHub에 올리기 | 포장한 짐을 창고(GitHub 서버)에 부치는 것 |

두 단계를 기억하세요.

```
[ 내 컴퓨터 ] → Commit (포장 + 라벨) → Push (창고에 올리기) → [ GitHub 서버 ]

"저장해줘"                              "올려줘"
```

Commit 없이 Push는 안 됩니다. 포장하지 않은 짐을 부칠 수 없는 것처럼요.

### Claude Code로 하면 이렇게 됩니다

명령어를 외울 필요가 없습니다. Claude Code에 한국어로 부탁하면 됩니다.

```
"이 폴더 변경사항 저장해줘. '3월 매출 분석 완료'라고 라벨 붙여서."
→ Claude Code가 알아서 git add + git commit 실행

"GitHub에 올려줘."
→ Claude Code가 알아서 git push 실행
```

이것이 이번 레슨의 핵심입니다. 터미널 git 명령어는 Claude Code가 대신합니다.

GitHub에 로그인한 상태에서:

1. 오른쪽 위 **"+"** 버튼 클릭 → **"New repository"** 선택

화면에 `https://github.com/사용자명/jiwoo-creator-workspace.git` 주소가 보이면 창고가 만들어진 겁니다. 이 주소를 복사해 둡니다.

### 6단계: 폴더 확인 + Git 초기화 + GitHub 연결

이제 폴더를 GitHub 창고에 연결합니다.

```text
안녕하세요! 저는 1인 콘텐츠 사업자 지우입니다.
이 폴더를 GitHub에 백업하려고 해요.
먼저 이 폴더에 어떤 파일들이 있는지 확인해 주고,
그 다음 이 폴더를 Git으로 초기화해서 방금 만든 GitHub 창고
https://github.com/내-사용자명/jiwoo-creator-workspace.git
에 연결해줘요. 단계별로 어떤 명령을 실행하는지 설명하면서 진행해줘요.
```

**기대 결과**:

먼저 폴더 파악 결과가 나옵니다.

```
이 폴더(practice-data/creator/)에는 다음 파일들이 있습니다:

- README.md                — 이 실습 폴더 안내 문서
- course_sales_monthly.csv — 강의별·월별 매출 데이터 (2025-07 ~ 2026-03)
- newsletter_stats.csv     — 뉴스레터 호별 발송·오픈율·클릭율 (18호 ~ 41호)
- student_inquiries.csv    — 수강생 문의 내역
- content_plan.md          — 2분기 콘텐츠 기획 노트
- research_memo.md         — AI 도구·1인 사업 시장 리서치 메모
- consulting_schedule.md   — 컨설팅·강의 운영 일정
```

그 다음 Git 초기화 과정이 진행됩니다.

```
1. git init
   → 이 폴더를 Git이 관리하는 공간으로 초기화합니다

2. git remote add origin https://github.com/...
   → GitHub 창고와 연결합니다

3. .gitignore 파일 생성
   → 민감 정보(API 키, .env 파일 등)가 실수로 올라가지 않도록 보호합니다
   추가된 항목: .DS_Store / .env / *.env / node_modules/ / .claude/
```

**관찰/체크포인트**:
- Claude Code가 "무슨 명령을 왜 실행하는지" 설명하면서 진행하는지 확인합니다.
- `.gitignore` 파일이 생성됐는지 확인하세요. `.env` 항목이 들어 있는지가 중요합니다. 이 파일이 없으면 API 키가 GitHub에 올라갈 수 있습니다. 확인 방법: Claude Code에게 "`.gitignore` 내용 보여줘"라고 요청하거나 터미널에서 `cat .gitignore`를 실행하면 됩니다.

> 💡 **안심 포인트**: Git 초기화는 폴더 안에 `.git`이라는 숨은 폴더를 만드는 것입니다. 기존 파일은 전혀 바뀌지 않습니다. 나중에 Git 관리를 그만두고 싶으면 이 `.git` 폴더만 삭제하면 됩니다.

### 7단계: 첫 번째 Commit + Push

GitHub 연결이 됐으니, 실제 파일을 올려봅니다.

```text
이 폴더의 모든 파일을 GitHub에 올리고 싶어요.
```

## 🔍 흔한 오류와 해결법

| 상황 | 원인 | 해결 방법 |
|------|------|----------|
| `cd` 후 `No such file or directory` | 폴더 경로가 다름 | `find ~ -name "course_sales_monthly.csv"` 로 실제 경로 찾고 그 경로로 이동 |
| `git: command not found` | Git이 설치 안 됨 | "Git 설치해줘"라고 Claude Code에 요청. Mac: `brew install git` / Ubuntu/WSL: `sudo apt install git` |
| `gh: command not found` | GitHub CLI 미설치 | "GitHub CLI 설치해줘"라고 요청. Mac: `brew install gh` |
| Push할 때 "Permission denied" 오류 | GitHub 인증이 안 된 상태 | `gh auth login` 실행 후 브라우저에서 인증 → 다시 시도 |
| Push할 때 "remote: Repository not found" | Repository 주소를 잘못 입력함 | "git remote -v 로 연결된 주소 확인해줘" → 주소 수정 후 재시도 |
| `error: failed to push some refs` | 원격에 더 최신 버전이 있음 | "원격에서 최신 내용 받아오고 다시 올려줘 (pull 후 push)" 요청 |
| 파일을 수정했는데 GitHub에 반영 안 됨 | Commit은 했는데 Push를 안 함 | "GitHub에 올려줘"라고 Claude Code에 다시 요청 |
| `git add` 후 "nothing to commit" 메시지 | 파일을 수정하지 않았거나 이미 Commit된 상태 | 정상 메시지입니다. 파일을 수정한 뒤 다시 시도 |
| `.env` 파일이 GitHub에 올라감 | .gitignore에 `.env`가 빠져 있음 | "`.env`를 .gitignore에 추가하고 이미 올라간 .env 파일은 삭제해줘" 요청 |
| `git restore` 후 아무 메시지 없음 | 정상 동작 — git restore는 성공해도 출력이 없는 경우가 많음 | `ls` 또는 파일 탐색기에서 파일이 다시 보이는지 확인 |
| Commit 메시지를 영어로 써야 하나요? | 관습이 영어일 뿐, 한국어도 됩니다 | "지우의 규칙서 초기 버전"처럼 한국어 메시지 완전히 괜찮습니다 |
| GitHub 접속은 되는데 파일이 안 보임 | Push가 아직 안 된 것 | "git status 확인해줘" 요청 후 상태 파악 |

## 💡 핵심 규칙·개념 정리

### Git 용어 한눈에

| 용어 | 우리말 | 비유 |
|------|--------|------|
| Repository (저장소) | GitHub의 내 프로젝트 폴더 | Google Drive의 공유 폴더 |
| Commit (커밋) | 저장하기 + 메모 달기 | 파일을 포장해서 라벨 붙이기 |
| Push (푸시) | GitHub에 올리기 | 포장한 짐을 창고로 발송 |
| Pull (풀) | GitHub에서 내려받기 | 다른 기기에서 최신본 가져오기 |
| .gitignore | 업로드 제외 목록 | "이 파일은 공유 폴더에 넣지 마" 메모 |
| Clone (클론) | 저장소 통째로 복사 | GitHub 저장소를 다른 컴퓨터에 내려받기 |

### 규칙 1: Commit은 세이브 포인트다

게임에서 세이브를 자주 해야 나중에 불상사가 생겨도 돌아갈 수 있습니다. Commit도 마찬가지입니다. 큰 작업을 한 번에 모아서 하지 말고, **의미 있는 단위로 자주** 하는 것이 좋습니다.

```
❌ 3주치 작업을 한 번에 "많은 것들 저장" 으로 Commit
✅ "3월 매출 분석 완료" / "뉴스레터 36호 분석" / "CLAUDE.md 규칙 3개 추가" 로 나눠서
```

### 규칙 2: Commit 메시지는 "미래의 나에게 쓰는 메모"

3개월 뒤에 히스토리를 볼 때, "이게 뭔 작업이지?" 싶지 않으려면 메시지가 구체적이어야 합니다.

```
❌ "수정함" / "업데이트" / "작업"
✅ "클로드코드 강의 3월 수강생 문의 패턴 분석 완료" / "뉴스레터 오픈율 하락 원인 정리"
```

### 규칙 3: Commit + Push는 세트

Commit(저장)만 하고 Push(올리기)를 안 하면 내 컴퓨터에만 있습니다. 작업이 끝나면 반드시 두 단계 모두.

### 규칙 4: Private Repository면 내 자료는 내 것

Private(비공개) Repository에 올린 파일은 나만 볼 수 있습니다. 고객 문의 내역, 매출 자료, 사업 계획 등 민감한 자료를 올려도 됩니다. 단, 비밀번호·API 키 같은 진짜 기밀 정보는 `.gitignore`로 보호하거나 애초에 파일에 넣지 않는 게 원칙입니다.

### 규칙 5: Claude Code가 Git 명령어를 대신한다

`git add`, `git commit`, `git push` 등 Git 명령어를 외울 필요 없습니다. Claude Code에 한국어로 부탁하면 됩니다. 이미 배운 cc-01~06의 "한국어로 시키면 된다"는 원칙 그대로입니다.
