# codex-09: 프롬프트 전략 + 고급 활용 — Codex를 제대로 부리는 사람들의 비밀

> 🔎 **이 레슨의 명령·모델명(gpt-5.x)·플래그·슬래시 명령은 실제 Codex 0.140.0 실측 + 공식 docs(`developers.openai.com/codex`) 재확인 예정 (2026-06 스냅샷 기준으로 작성됨)**

> **레슨 목표**: Codex에게 훨씬 좋은 결과를 뽑아내는 프롬프트 전략(outcome-first)을 익히고, `/plan`·`@파일` 고급 기능과 자연어 검토 요청을 지우의 실제 데이터(매출·문의·뉴스레터)로 직접 써본다.
> **예상 시간**: 60분 (개념 도입 10분 + 실습 1 20분 + 실습 2 25분 + 실습 3 선택 10분 + 마무리 5분)
> **사전 준비**: Codex 설치·로그인 완료(codex-01) / `practice-data/creator/` 폴더 열어 두기 / 이전 모듈(codex-01~08) 수강 완료
> **진입 커맨드**: `/codex-09`

## 📚 학습 목표 — 4가지 와우 모먼트

1. **"같은 부탁인데 결과가 왜 이렇게 달라?" 체험** — 프롬프트(요청문)의 구조가 답의 품질을 결정한다는 것을 몸으로 안다
2. **outcome-first 프롬프트를 손에 익힌다** — "목표 + 성공 기준 + 제약"을 붙이면 반복 수정이 확 줄어든다는 것을 직접 비교로 확인한다
3. **`/plan`·`@파일` 고급 기능과 자연어 검토 요청을 실전에서 써본다** — 이것만 알면 Codex 활용도가 두 배 이상 뛴다
4. **지우의 실제 데이터(매출·문의·뉴스레터)로 분기 보고서 초안까지 완성한다** — 연습이 아니라 내 사업에 바로 쓸 수 있는 산출물을 손에 쥔다

## 🎯 개념 도입 — "어떻게 부탁하느냐"가 전부다

### 비유 — 식당에서 주문하는 두 방법

같은 식당, 같은 요리사인데 결과가 완전히 달라지는 상황을 생각해봅시다.

**방법 A**: "뭔가 맛있는 거 주세요."
→ 요리사는 그날 재료 중 자기가 좋아하는 걸 냅니다. 당신이 원한 게 아닐 수도 있어요.

**방법 B**: "4명이서 먹을 건데, 매운 거 못 먹는 사람이 있어요. 예산은 10만 원이고, 오늘은 고기 쪽이 좋겠어요. 1인 1접시로 나눠주세요."
→ 요리사가 정확히 원하는 방향으로 음식을 준비합니다.

Codex에게 일을 시킬 때도 똑같습니다.

**막연한 요청** (방법 A):
```text
student_inquiries.csv 분석해줘
```
무엇을 분석할지, 어떤 형태로 줄지, 무엇이 중요한지 — 모두 Codex의 추측에 맡겨집니다. 결과가 나오긴 하지만, 내가 원하는 형태가 나올 확률은 낮습니다.

**outcome-first 요청** (방법 B):
```text
@student_inquiries.csv 에서 카테고리별 문의 건수를 세서 건수 많은 순으로 정렬한 표를 만들어줘.
"실습" 카테고리 문의는 내용을 요약해서 따로 정리해줘 — 비개발자가 자주 막히는 구간을 찾는 게 목적이야.
처리 중(진행중) 상태인 문의가 몇 건인지도 알려줘.
마지막에 "다음에 취할 행동 1가지"를 한 줄로 제안해줘. 한국어로.
```

### outcome-first 패턴의 세 재료

좋은 요청에는 이 세 가지가 들어갑니다.

```
목표(Goal)          — 무엇을 원하나 (표? 목록? 보고서? 초안?)
성공 기준(Done)     — 어떻게 됐을 때 끝난 건지 ("5분 안에 읽을 수 있게", "다음 행동 포함")
제약(Constraints)   — 이건 하지 마 / 이것만 봐 / 이 형식으로
```

기억하기 쉽게 줄이면: **목표 + 끝나는 기준 + 조심할 것**.

### 두 가지 검토 도구 미리 알기

이번 레슨에서 핵심적으로 쓸 두 가지입니다.

> ⚠️ `/plan`은 이 레슨(codex-09)에서 처음 소개하는 고급 기능입니다. `/review`는 코드·변경사항(diff) 리뷰 전용 슬래시 명령으로, 분석 결과 검토는 자연어로 요청하는 것이 정확합니다. 아래 설명은 2026-06 스냅샷 기준입니다.

| 도구 | 언제 쓰나 | 효과 |
|------|---------|------|
| `/plan` | 여러 파일을 건드리거나 복잡한 작업 전 | 계획을 먼저 보고 방향 확인, 오해 없이 시작 |
| `/review` | 코드·변경사항(diff) 리뷰 시 | Codex가 코드 변경이나 diff를 검토하고 문제점을 찾아줌 |

**비유**:
- `/plan` = 인테리어 업체에 "우리 집 바꿔주세요" 하기 전에 "설계도 먼저 보여줘요"
- `/review` = 코드 수정 후 "이 변경사항 검토해줘" (코드/diff 전용)

### 세 가지 고급 기능 한눈에

| 기능 | 무엇인가 | 언제 쓰나 |
|------|---------|---------|
| **`@파일명`** | 특정 파일을 정확히 지목해서 읽힘 | 여러 파일 중 하나만 집어서 줄 때 |
| **`/plan`** | 실행 전 단계별 계획을 먼저 보여줌 | 복잡·중요한 작업 전 방향 확인할 때 |
| **`/review`** | 코드·변경사항(diff)을 검토 | 코드를 수정하거나 변경사항 리뷰가 필요할 때 |

> 💡 **`@파일명` 문법**: `@student_inquiries.csv`처럼 파일 앞에 `@`를 붙이면 "이 파일을 정확히 집어서 봐"라고 지목합니다. 폴더에 파일이 여러 개 있을 때 특히 유용합니다.

### Codex만의 강력한 기능 3가지 더 — 서브에이전트·이미지·목표 모드

Codex에는 위 도구 외에도 일을 크게 키워주는 기능이 있습니다. (지금 다 쓸 필요는 없어요 — "이런 것도 되는구나" 알아두면 나중에 큽니다.)

| 기능 | 무엇인가 | 어떻게 부르나 |
|------|---------|--------------|
| **서브에이전트** | 보조 일꾼 여럿을 동시에 굴려 한꺼번에 처리 | 자연어로 "~관점으로 하나씩 띄워서 정리해줘" |
| **이미지 생성** | 썸네일·일러스트 같은 그림을 직접 생성 | `$imagegen` 또는 자연어 |
| **`/goal` (목표 모드)** | 목표만 주면 조건 충족까지 알아서 진행 | `/goal <조건>` |

**① 서브에이전트 — "보조 일꾼 여럿을 동시에"** (cc-10에서 본 Claude Code 서브에이전트와 같은 개념)

```text
이 폴더의 자료를 병렬로 검토해줘. 매출 관점, 고객 문의 관점, 콘텐츠 관점으로
서브에이전트(보조 일꾼)를 하나씩 띄우고, 다 끝나면 결과를 표 하나로 정리해줘.
```
혼자 순서대로 하는 것보다, 갈래가 분명한 큰 일을 **나눠서 동시에** 할 때 빠릅니다. (작은 일은 그냥 한 명이 나아요.)

**② 이미지 생성 — "그림도 직접"**

```text
$imagegen 흰 배경에 두부 마스코트 일러스트 한 장 만들어줘. 글자 없이 PNG로.
```
강의 썸네일, 뉴스레터 삽화 같은 걸 Codex가 바로 만들어 줍니다. (기본 그림 생성은 별도 키 설정 없이 됩니다.)

**③ `/goal` — "이 목표까지 알아서"**

```text
/goal 이번 분기 매출·문의 데이터를 분석해서 요약 보고서를 만들고,
마지막에 남은 위험·확인할 점을 알려줘
```
- `/goal` = 현재 목표 확인 · `/goal pause` / `/goal resume` = 잠깐 멈춤·재개 · `/goal clear` = 취소
- 목표(끝나는 조건)를 주면 그때까지 스스로 여러 단계를 진행합니다.

> ⚠️ **자율 기능은 멈추는 법부터**: `/goal`처럼 알아서 가는 기능은 엉뚱하면 `Esc`(또는 `/goal clear`)로 멈춥니다. 처음엔 작은 일로, 중요한 변경은 지켜보며 쓰세요.

> 🟢 이 셋은 부록 중에서도 "더 보기"입니다. 당장 안 써도 되고, 나중에 "이런 게 있었지" 하고 꺼내 쓰면 됩니다.

## 🔧 실습 1 — "같은 요청, 다른 프롬프트" 직접 비교하기

이 실습의 핵심은 **같은 데이터로 막연한 요청과 outcome-first 요청을 차례로 써보고, 결과의 차이를 눈으로 확인하는 것**입니다.

사용 파일: `practice-data/creator/student_inquiries.csv`

이 파일에는 2026년 3월 첫 주 수강생 문의 24건이 담겨 있습니다. 채널(이메일·카톡·인스타DM), 카테고리(결제·실습·환불·수강방법·수료증·강의문의), 처리상태, 소요시간 등이 포함되어 있으며, 현재 '진행중' 상태인 문의가 5건입니다.

---

### 1단계: Codex 세션 열기

터미널을 열고 `creator` 폴더를 작업 대상으로 지정하며 Codex를 시작합니다.

```bash
codex --cd ~/path/to/practice-data/creator
```

> 💡 `~/path/to/` 부분은 실제 파일이 있는 경로로 바꿔주세요. 예: `codex --cd ~/Documents/cc-codex-101/practice-data/creator`. 현재 위치를 모르겠으면 `pwd`를 입력하면 됩니다. `codex --cd 경로` 단독 입력으로 대화 세션이 열립니다 — 여기서 원하는 작업을 직접 입력하면 됩니다.

**기대 결과**: Codex가 실행되고 입력 대기 상태(프롬프트)가 열립니다.

**체크포인트**: Codex 입력창이 열리면 성공입니다. 아무 메시지도 없어도 정상입니다.

---

### 2단계: 막연한 요청으로 먼저 시도하기

아래를 그대로 입력해봅니다.

```text
student_inquiries.csv 분석해줘
```

**기대 결과 (예시)**:
```
student_inquiries.csv 파일을 읽었습니다.
총 24건의 문의가 있으며, 카테고리는 결제·환불·실습·수강방법·수료증·강의문의로 나뉩니다.
실습 관련 문의가 가장 많으며, 처리 완료된 건과 진행 중인 건이 섞여 있습니다.
```

**관찰 포인트**: 결과가 나오긴 했지만 — 어떤 형태로 정리됐나요? 표가 나왔나요, 목록이 나왔나요? 다음 행동이 포함됐나요? 잠깐 살펴보고 기억해 두세요.

### 3단계: outcome-first 프롬프트로 같은 데이터에 다시 시도하기

이번엔 목표·형태·성공 기준을 명시해서 다시 요청합니다.

```text
@student_inquiries.csv 에서:
1) 카테고리별 문의 건수를 센 후 건수 많은 순으로 정렬한 표를 만들어줘
2) "실습" 카테고리의 문의 내용을 요약해줘 — 비개발자가 자주 막히는 구간을 찾는 게 목적이야
3) 처리 중(진행중)으로 남아있는 문의가 몇 건인지, 누구인지 알려줘
4) 마지막에 "이 데이터를 보고 다음에 취할 행동 1가지"를 한 줄로 제안해줘
결과는 비개발자가 읽기 쉬운 한국어로.
```

**기대 결과 (예시)**:
```
📊 카테고리별 문의 현황 (2026년 3월, 총 24건)

| 카테고리 | 건수 |
|---------|------|
| 실습     | 7건  |
| 결제     | 4건  |
| 강의문의 | 5건  |
| 환불     | 3건  |
| 수강방법 | 3건  |
| 수료증   | 2건  |

🔍 실습 문의 요약 (7건)
비개발자가 자주 막히는 구간:
- 터미널 명령어 에러 (맥, 정유나): 2026-03-02 진행중
- AGENTS.md 파일 위치 모름 (윤서아): 완료
- WSL 설치 어려움 (배준혁): 2026-03-05 진행중
- CLAUDE.md 규칙 미적용 (조은서): 2026-03-06 진행중
- 실습 데이터 파일 위치 (유하린): 완료
- Codex 로그인 자꾸 풀림 (백서윤): 2026-03-08 진행중
- 프롬프트 결과 불일치 (임채원): 완료
→ 공통점: 대부분 "설치·환경 설정" 단계에서 막힘. 특히 WSL(윈도우용) 설치 반복 문의.

⚠️ 처리 중(진행중) 문의: 5건
이도윤(환불), 정유나(터미널 실습), 배준혁(WSL 실습), 조은서(CLAUDE.md), 백서윤(Codex 로그인)

💡 다음 행동 제안:
WSL 설치 문의가 반복되고 있어(배준혁·백서윤 모두 진행중), "윈도우 WSL 설치 단계별 안내 영상"
하나를 만들어 자동 안내로 돌리면 이후 동일 문의를 크게 줄일 수 있습니다.
```

**관찰 포인트**:
- 2단계 결과와 비교해보세요 — 어느 쪽이 더 바로 쓸 수 있나요?
- `@파일` 문법을 써서 Codex가 그 파일만 정확히 읽었나요?
- "다음 행동 1가지" 제안이 들어갔나요?

### 4단계: 차이를 자기 말로 정리하기

잠깐 멈추고, 두 결과의 차이를 자신의 말로 한 줄 적어봅니다.

예: "목표·기준·조심할 것을 붙이니까 Codex가 내가 원하는 형태 그대로 답했고, 다음 행동까지 나왔다."

> 💡 이 차이를 느끼는 것 자체가 이 레슨의 핵심 와우 모먼트입니다.

---

### 5단계: `/plan`으로 계획 먼저 확인하기

복잡한 작업 전에 `/plan`을 써서 Codex가 어떻게 진행할지 먼저 보는 방법을 체험합니다.

```text
/plan student_inquiries.csv에서 문의 유형별 건수와 평균 처리 시간을 정리하고, 처리 시간이 가장 긴 유형과 짧은 유형을 비교해서 보고서 초안을 만들어줘.
```

**기대 결과 (예시)**:
```
계획:
1. student_inquiries.csv 파일을 읽고 열 구조를 파악합니다.
2. '카테고리' 열 기준으로 문의 유형별 건수를 집계합니다.
3. '소요시간(시간)' 열 기준으로 유형별 평균을 계산합니다.
4. 처리 시간이 가장 긴 유형 / 짧은 유형을 비교합니다.
5. 결과를 보고서 초안 형태로 정리합니다.

이렇게 진행할까요?
```

**관찰 포인트**: "이렇게 진행할까요?" 하고 묻습니다. 방향이 맞으면 "네, 진행해줘"라고 하면 됩니다. 방향이 다르면 "2단계는 채널별로 해줘"처럼 수정 후 진행합니다. 간단한 작업에서는 `/plan` 없이 바로 진행하기도 하므로, 여러 파일을 연결하거나 보고서처럼 복잡한 작업일 때 특히 유용합니다.

**체크포인트**: `/plan`을 쓰면 계획을 먼저 보여주고 승인을 구하는 흐름을 확인했습니다.

---

### 6단계: 승인 후 진행 + 결과 검토 체험하기

계획을 확인하고 승인합니다.

```text
네, 진행해줘.
```

Codex가 작업을 완료하면, 결과를 받은 직후 바로 자연어로 검토를 요청합니다.

```text
방금 만든 결과에서 빠진 내용이나 보완하면 좋을 점이 있으면 알려줘.
```

**기대 결과 (예시)**:
```
검토 결과:
- 문의 유형 6가지 모두 반영됨 ✅
- '실습' 유형의 평균 처리 시간(1.7시간)이 가장 길고, '수료증'(0.25시간)이 가장 짧음 ✅
- 보완 가능한 점: 강의명별 문의 분포가 없습니다. 어떤 강의에 실습 지원이 집중되는지
  추가하면 보고서가 더 완전해집니다. 추가할까요?
```

**관찰 포인트**: 자연어 검토 요청은 "잘 됐어요"가 아니라 빠진 부분까지 스스로 찾아 제안합니다. "네, 추가해줘"라고 하면 보완됩니다. 이것이 "한 번 더 확인시키는 법"입니다. 더 구체적으로 지목하고 싶으면 "어떤 점이 부족할 수 있는지 특히 찾아봐줘"라고 추가하면 됩니다.

> 💡 **참고**: `/review`는 코드 변경(diff) 리뷰 전용 슬래시 명령입니다. 분석 결과·보고서 검토는 위처럼 자연어("방금 결과 검토해줘")로 요청하는 것이 정확합니다.

## 🎬 실습 2 — 지우의 사업 자료로 분기 보고서 한 사이클 돌리기

### 2단계: 매출 데이터 핵심 패턴 찾기


### 7단계: `/compact`로 대화 정리하기

실습을 하다 보면 대화가 길어집니다. 큰 분석 하나가 끝났을 때:

## 💡 핵심 규칙 / 개념 정리

### Codex에게 잘 부탁하는 법 — 3가지 원칙

**원칙 1: outcome-first (결과부터 명시)**
```
[무엇을 원하나] + [이렇게 되면 끝] + [이건 하지 마]
```

```
❌ 막연한 요청: "매출 분석해줘."

✅ outcome-first: "@course_sales_monthly.csv 에서 2026년 1~3월 강의별 순매출을
   높은 순으로 표로 보여주고, 전월 대비 증감률도 붙여줘.
   마지막에 집중할 강의 1개 추천을 이유와 함께 한 줄로. 한국어로."
```

**원칙 2: `@파일`로 정확히 지목하기**

폴더에 파일이 여러 개 있을 때 `@파일명`을 쓰면 "이 파일만 봐"라고 정확히 지목합니다. 두 파일을 연결할 땐 `@파일1 와 @파일2 를 같이 읽어줘`.

**원칙 3: 검토를 시켜라**

결과가 나오면 "방금 결과 검토해줘" 또는 "빠진 점 있으면 알려줘"로 한 번 더 확인시키세요. 중요한 수치는 항상 원본 파일과 교차 확인합니다.

### 자주 쓸 세 가지 흐름

```
복잡한 작업: /plan → 계획 확인·수정 → 승인 → 진행 → "방금 결과 검토해줘"
일반 작업:   outcome-first 요청 → 진행 → "빠진 점 있으면 알려줘" (선택)
파일 지정:   @파일이름으로 정확히 지목 (두 파일은 @파일1 와 @파일2)
```

### 프롬프트 품질 자가점검 (발송 전 5초 확인)

- [ ] 어떤 파일을 봐야 하는지 명확한가? (`@파일명` 썼는가)
- [ ] 결과물의 형태(표, 문장, 목록)를 말했는가?
- [ ] "끝났다"의 기준을 말했는가?
- [ ] 언어(한국어)를 지정했는가?
- [ ] 복잡한 작업이면 `/plan`을 먼저 쓸 것인가?

### 과지정(over-specification)을 조심하세요

outcome-first가 중요하지만, 너무 세세하게 다 정해주면 Codex가 정작 잘하는 부분(스스로 방향 잡기)을 발휘하지 못합니다. "목표 + 성공 기준 + 제약" 정도면 충분합니다. 나머지는 Codex에게 맡기세요.
