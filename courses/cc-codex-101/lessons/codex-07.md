# codex-07: MCP 외부도구 연결 — "AI 비서에게 바깥 연장을 쥐여주는 법"

> **레슨 목표**: Codex에 MCP 서버를 연결해, 내 사업 데이터(폴더 안)와 외부 정보(웹·공식 문서)를 같은 대화 안에서 함께 다루는 법을 익힌다.
> **예상 시간**: 50분 (개념 10분 + 실습 1 연결 15분 + 실습 2 실전 분석 20분 + 실습 3 선택 5분)
> **사전 준비**: Codex 설치·로그인 완료(`codex login` 통과) / `AGENTS.md` 기본 작성 완료(codex-06 이수) / `practice-data/creator/` 폴더 접근 가능 / 터미널 1개 열어두기
> **진입 커맨드**: `/codex-07`
>
> 🔎 **검증 완료(codex-cli 0.140.0 실측)**: 이 레슨의 명령·모델명(gpt-5.x)·플래그·MCP 서버 설치 경로는 **실제 Codex 세션(0.140.0) 실측 + 사용 직전 공식 docs(`developers.openai.com/codex`) 재확인 예정(2026-06 스냅샷 기준)**. 불확실한 구체값은 codex-101-정리.md 값만 사용하며, MCP 명령(codex mcp add/list/remove)·workspace-write 샌드박스 정확 확인 완료.

## 📚 학습 목표 — 4가지 와우 모먼트

1. **"Codex가 웹도 볼 수 있게 된다"** — MCP를 연결하면 Codex의 능력이 내 파일을 넘어 외부 세계로 확장된다는 것을 체감한다.
2. **"연결 명령 딱 한 줄이면 된다"** — 복잡해 보이는 외부 도구 연결이 실제로는 매우 단순하다는 자신감을 얻는다.
3. **"CC에서 배운 MCP 개념이 Codex에서도 그대로다"** — 같은 표준을 쓰는 두 도구의 공통점을 직접 확인해, 다음 코스에서 두 도구를 한 팀처럼 쓰는 토대를 쌓는다.
4. **"내 파일 분석 + 웹 정보를 한 대화에서"** — `course_sales_monthly.csv`, `newsletter_stats.csv`, `student_inquiries.csv` 분석에 MCP가 어떻게 끼어드는지 체감하고, "AI가 바깥도 본다"는 감각을 익힌다.

## 🎯 개념 도입 — "콘센트를 꽂아야 바깥 세상과 연결된다"

### 지금 Codex는 어떤 상태일까

지우가 지금까지 쓰던 Codex를 상상해 보자.

> 아주 똑똑한 비서가 지우의 사무실 책상 앞에 앉아 있다. 그런데 이 비서는 **책상 위에 있는 파일만** 볼 수 있다. 인터넷 검색? 안 된다. 경쟁 강의 플랫폼에서 최신 정책 확인? 불가. Codex 공식 문서를 직접 찾아보는 것? 그것도 안 된다.

이유는 **샌드박스(안전 울타리)** 때문이다. 샌드박스는 지우의 컴퓨터를 안전하게 지키는 장치인데, 기본값으로는 인터넷 연결도 막혀 있다. codex-05에서 배웠듯, `workspace-write` 샌드박스는 내 폴더 안에서는 읽고·쓰고·실행하지만 **네트워크는 기본 차단**이다.

**MCP(Model Context Protocol, '모델-도구 연결 표준')** 는 이 벽을 허무는 방법이다.

> **비유**: MCP는 콘센트다.
> - 비서(Codex)의 책상에 콘센트 구멍이 여러 개 있다.
> - '파일 검색' 플러그, '웹 브라우저' 플러그, 'Codex 공식 문서' 플러그를 하나씩 꽂으면 그 순간부터 비서가 그 도구를 쓸 수 있게 된다.
> - 꽂지 않으면 없는 것과 같다. 꽂으면 그 순간부터 쓸 수 있다.

이게 전부다. 복잡한 개념이 아니라, **"필요한 도구를 꽂는다"** 는 단순한 작업이다.

---

### MCP가 없을 때와 있을 때

| 상황 | MCP 없음 | MCP 있음 |
|---|---|---|
| 최신 Codex 사용법 확인 | 기억 속 정보만(구버전일 수 있음) | openaiDeveloperDocs → 실시간 공식 문서 조회 |
| 경쟁 강의 리서치 | 지우가 직접 검색 → 복사·붙여넣기 | Playwright(웹 브라우저) → Codex가 직접 찾아옴 |
| 폴더 안 파일 분석 | 가능(항상 됐음) | 더욱 풍부하게 — 파일 + 웹 정보 동시에 |
| 노션·슬랙 데이터 연동 | 불가 | 해당 MCP 서버 연결 후 가능 |

지우에게 지금 당장 가장 유용한 건 두 가지다.
- **Codex 공식 문서 MCP**: Codex 사용법이 헷갈릴 때 Codex 스스로 최신 문서를 찾아봐 주는 것
- **웹 브라우저 MCP(Playwright)**: 경쟁 강의 리서치, 트렌드 파악, 플랫폼 정책 확인

---

### CC와 Codex의 MCP — 같은 표준

cc-09에서 MCP를 배웠거나 배울 예정이라면, 핵심 원리는 그대로 옮겨진다.

| 항목 | Claude Code | Codex |
|---|---|---|
| MCP 표준 | 동일 — 완전히 같은 표준 | 동일 — 완전히 같은 표준 |
| 연결 명령 | `claude mcp add ...` | `codex mcp add ...` |
| 설정 저장 | `~/.claude/` 또는 프로젝트 설정 | `config.toml` `[mcp_servers.*]` |
| 연결된 도구 쓰기 | 세션에서 자동 인식·사용 | 동일 |

**한 줄 요약**: "플러그 규격은 두 도구가 완전히 같다. 꽂는 손(명령어)만 `claude`에서 `codex`로 바뀐다."

---

### 왜 '부록'인가 — "처음부터 다 연결할 필요는 없다"

codex-07이 부록으로 분류된 이유가 있다. MCP는 강력하지만, 처음부터 모든 기능을 다 연결할 필요는 없다.

비유하면 이렇다. 처음 스마트폰을 사면 기본 앱만으로도 전화·문자·카메라는 다 된다. 앱스토어에서 앱을 더 깔면 더 많은 일을 할 수 있지만, 처음부터 100개 앱을 깔 필요는 없다. **필요할 때 하나씩.**

MCP도 마찬가지다. 기본 Codex만으로도 지금까지 해온 파일 분석·보고서 초안·문의 정리는 충분히 된다. MCP는 "한 단계 더"가 필요할 때 붙이는 확장팩이다.

## 🔧 실습 1 — MCP 서버 연결하기 (15분)

이 실습에서는 **OpenAI 개발자 문서 MCP**와 **Playwright(웹 브라우저) MCP**를 연결해본다. 지우가 가장 자주 쓸 두 도구다.

### 1단계: 현재 MCP 연결 상태 확인

Codex 터미널에서 시작하기 전, 먼저 지금 상태를 확인한다. "아무것도 없는 상태"를 눈으로 보고 시작해야 "연결됐다"는 차이가 선명하게 보인다.

```text
codex mcp list
```

**기대 결과**:
```
No MCP servers configured.
```

아직 아무것도 연결하지 않았으면 이렇게 나온다. 이미 뭔가 연결된 목록이 있어도 괜찮다 — 그냥 현재 상태 확인이 목적이다.

**관찰 포인트**: "비서 책상에 콘센트 플러그가 하나도 없는 상태"다. 이제 하나씩 꽂아보자.

---

### 2단계: OpenAI 개발자 문서 MCP 연결

Codex 공식 문서를 Codex가 직접 찾아볼 수 있게 연결한다. 지우가 Codex 사용법이 헷갈릴 때 Codex 스스로 최신 문서를 참조해 답해준다.

```text
codex mcp add openaiDeveloperDocs --url https://developers.openai.com/mcp
```

**기대 결과**:
```
✓ MCP server 'openaiDeveloperDocs' added successfully.
```

**관찰 포인트**:
- 한 줄로 끝났다. 설치가 아니라 "연결"이라 빠르다.
- `--url` 뒤가 MCP 서버 주소다. 누군가 이미 만들어둔 서버에 Codex가 접속하는 방식이다.

---

### 3단계: MCP 연결 명령의 구조 이해하기

실제 연결 전에 명령의 구조를 잠깐 살펴보자. 이해하면 나중에 다른 MCP도 겁 없이 연결할 수 있다.

```text
codex mcp add 도구이름 --url 서버주소
codex mcp add 도구이름 -- npx 패키지이름
```

- `codex mcp add` → "MCP 도구를 하나 추가해"
- `도구이름` → 내가 부를 별명(자유롭게 정하면 됨)
- `--url ...` 또는 `-- npx ...` → 실제로 어디서 이 도구를 가져오는지

**비유**: 도서관에서 책을 빌릴 때 "카드 제시 → 책 번호 → 대출 완료"인 것처럼, MCP도 `codex mcp add → 도구 이름 → 어디서 가져올지` 세 단계다. 외울 필요 없이 필요할 때 공식 문서나 MCP 제공자 페이지에서 복사하면 된다.

---

### 4단계: Playwright(웹 브라우저) MCP 연결

Playwright는 Codex가 직접 웹 브라우저를 열어 정보를 찾아오게 해주는 도구다. 강의 플랫폼 정책, 경쟁 강의 정보, 트렌드 리서치를 Codex가 직접 검색하게 만들 수 있다.

```text
codex mcp add playwright -- npx --yes @playwright/mcp@latest
```

**기대 결과**:
```
✓ MCP server 'playwright' added successfully.
```

**관찰 포인트**:
- `npx --yes @playwright/mcp@latest` 부분은 "Playwright MCP 서버 최신 버전을 자동 설치해서 쓴다"는 뜻이다.
- 처음 실행 시 패키지 다운로드로 1~2분 걸릴 수 있다. 인터넷이 연결되어 있으면 자동으로 완료된다.
- 정확한 플래그·버전은 실제 0.140.0 실측(codex-cli 0.140.0)으로 확정 완료.

---

### 5단계: 연결 후 목록 다시 확인

```text
codex mcp list
```

**기대 결과**:
```
MCP Servers:
  • openaiDeveloperDocs  →  https://developers.openai.com/mcp
  • playwright           →  npx @playwright/mcp@latest
```

**관찰 포인트**: 아까 "No MCP servers"였던 자리에 두 개가 추가됐다. 콘센트에 플러그 두 개가 꽂혔다.

---

### 6단계: config.toml에 영구 저장 확인

연결한 MCP는 `config.toml`에 자동으로 저장된다. Codex를 껐다 켜도 유지된다. codex-06에서 만든 그 설정 파일이다.

```text
cat ~/.codex/config.toml
```

**기대 결과**:
```toml
model = "gpt-5.5"
model_reasoning_effort = "medium"
approval_policy = "on-request"

[mcp_servers.openaiDeveloperDocs]
url = "https://developers.openai.com/mcp"

[mcp_servers.playwright]
command = "npx"
args = ["--yes", "@playwright/mcp@latest"]
```

**관찰 포인트**: `[mcp_servers.openaiDeveloperDocs]`처럼 MCP 설정이 codex-06에서 만든 기존 설정(model, effort, approval_policy) 아래에 자동 추가됐다. **`AGENTS.md`가 "내가 하는 일의 규칙"이라면, `config.toml`은 "Codex 동작의 기계 설정값"이다.** 규칙서와 설정 파일, 이 두 파일을 함께 관리하는 것이 안정적인 Codex 운용의 핵심이다.

---

### 7단계: MCP 연결 첫 테스트

연결된 MCP를 바로 써보자. Codex에게 간단한 질문을 던져 MCP가 실제로 작동하는지 확인한다.

Codex 세션 안에서 입력한다.

```text
openaiDeveloperDocs MCP를 써서, Codex의 AGENTS.md에 넣으면 좋은 섹션 구성을 알려줘. 비개발자 1인 사업자 기준으로 간단하게.
```

**기대 결과**:
```
[openaiDeveloperDocs 검색 중...]

AGENTS.md 권장 섹션 (비개발자 1인 사업자 기준):

1. 프로젝트 소개 — 내 사업이 무엇인지 (Codex가 맥락을 잡기 위해)
2. Codex의 역할 — 반복 작업 목록, 하지 말아야 할 작업 목록
3. 규칙 — 보고서 언어(한국어), 고객정보 보호, 파일 삭제 금지
4. 완료 기준 — 분석 결과엔 반드시 "다음에 뭘 하면 되는지" 1줄 포함

(실제 출력은 Codex 세션에 따라 다를 수 있다)
```

**관찰 포인트**: "openaiDeveloperDocs MCP를 써서"라고 명시적으로 요청했다. Codex가 내 파일 외에도 외부 공식 문서를 실시간으로 가져온다는 것을 처음 체감하는 순간이다.

## 🎬 실습 2 — MCP 연결 상태에서 지우의 사업 데이터 분석하기 (20분)

지우의 `practice-data/creator/` 실제 데이터를 MCP 연결 상태에서 분석한다. 파일을 읽는 것을 넘어, 외부 맥락과 결합하는 경험이 목표다.

### 1단계: 작업 폴더 지정 + 파일 탐색

```text
codex --cd ~/practice-data/creator
```

Codex 세션이 열리면 아래를 입력한다.

```text
이 폴더에 있는 파일 목록을 보여줘. 나는 1인 콘텐츠 강의 사업자야. 어떤 파일부터 보면 좋을지 우선순위도 알려줘.
```

**기대 결과**:
```
이 폴더의 파일:
1. course_sales_monthly.csv — 강의별·월별 매출 (9개월치, 2025-07~2026-03)
2. newsletter_stats.csv — 뉴스레터 18호~41호 성과
3. student_inquiries.csv — 수강생 문의 내역 24건
4. content_plan.md — 2분기 콘텐츠 기획 노트
5. research_memo.md — AI 도구·시장 리서치
6. consulting_schedule.md — 3월 운영 일정

추천 순서:
1. course_sales_monthly.csv — 사업 매출 기반 강의 성과 파악 (핵심)
2. newsletter_stats.csv — 콘텐츠 반응 패턴 (마케팅 효과 파악)
3. student_inquiries.csv — 수강생 목소리 (상품 개선 단서)
```

**관찰 포인트**: Codex가 폴더를 직접 탐색해 파일 역할을 파악하고 우선순위까지 제안한다. 지우가 폴더를 가리켰을 뿐인데 파일 하나하나를 직접 열지 않아도 된다.

---

### 2단계: 강의 매출 데이터 패턴 찾기

`course_sales_monthly.csv`에는 2025년 7월부터 2026년 3월까지 6개 강의의 월별 매출 데이터가 있다.

```text
@course_sales_monthly.csv 이 파일에서:
1. 강의별 누적 순매출 합계를 내림차순으로 정리해줘
2. 2026년 1~3월 최근 흐름을 월별로 보여줘
3. 가장 빠르게 성장한 강의가 뭔지 찾아줘
표 형식으로 정리해줘.
```

**기대 결과**:
```
1. 강의별 누적 순매출 (내림차순, 단위: 만원)

| 순위 | 강의명 | 카테고리 | 누적 순매출 |
|---|---|---|---|
| 1위 | 챗GPT로 콘텐츠 10배 빠르게 | AI활용 | 7,260 |
| 2위 | 1인 지식사업 수익화 로드맵 | 비즈니스 | 5,312 |
| 3위 | 노션으로 1인 사업 시스템 만들기 | 생산성 | 1,920 |
| 4위 | 클로드코드로 업무 자동화 입문 | AI활용 | 1,750 |
| 5위 | 초보 뉴스레터 7일 완성 | 글쓰기 | 1,268 |
| 6위 | 크리에이터 세금 기초 | 비즈니스 | 191 |

2. 2026년 1~3월 월별 흐름 (순매출 합산)
```

## 💡 핵심 규칙 / 개념 정리

### MCP란

**MCP(Model Context Protocol)** = AI 도구와 외부 세계를 잇는 연결 표준. Codex가 기본값으로 접근하지 못하는 외부 도구(웹, 다른 서비스, 공식 문서)를 연결하는 방법이다. CC와 Codex가 같은 MCP 표준을 공유한다 — 한 번 개념을 익히면 두 도구에 모두 통한다.

### MCP의 3가지 원칙

1. **필요할 때 하나씩**: 처음부터 모든 MCP를 연결할 필요 없다. 하고 싶은 일이 생겼을 때 그에 맞는 도구를 하나씩 붙인다.
2. **연결하면 영구 기억**: `config.toml`에 저장되면 다음에 Codex를 실행할 때 자동으로 연결된다. 단골 식당의 단골 주문처럼.
3. **CC와 개념 동일, 명령만 다름**: CC에서는 `claude mcp add 도구이름 ...`, Codex에서는 `codex mcp add 도구이름 ...`. 앞 단어만 바뀌었다.

### 핵심 명령 4개

```bash
codex mcp add 이름 --url 주소          # URL 방식 서버 연결
codex mcp add 이름 -- npx 패키지명    # 로컬 실행 방식 서버 연결
codex mcp list                         # 연결된 MCP 목록 확인
codex mcp remove 이름                  # 연결 해제
```

### 지우에게 유용한 MCP 후보

| MCP 서버 | 언제 쓰나 | 지우의 활용 예 |
|---|---|---|
| **openaiDeveloperDocs** | Codex 사용법 헷갈릴 때 | AGENTS.md 구성 조언, 명령 확인 |
| **playwright (웹 브라우저)** | 최신 정보·경쟁사·트렌드 | 강의 플랫폼 정책, 경쟁 강의 리서치 |
| Notion MCP (선택) | 노션에 정리된 자료 읽기/쓰기 | 기획 노트·일정을 Codex가 직접 참조 |
| Google Drive MCP (선택) | 구글 시트·문서 읽기 | 실제 매출 시트 연결 |

### 설정 파일 두 개의 역할 — 다시 정리

| 파일 | 역할 | 예시 내용 |
|---|---|---|
| **AGENTS.md** | 지우가 하는 일의 규칙 | 역할, 금지사항, 보고 형식, 완료 기준 |
| **config.toml** | Codex 동작의 기계 설정값 | 모델, effort, 승인 방식, MCP 서버 목록 |

두 파일 모두 "한 번 써두면 매번 같은 설정이 자동 적용"된다는 점에서 같은 철학이다. `AGENTS.md`는 사람이 읽는 규칙서고, `config.toml`은 기계가 읽는 설정값이라고 생각하면 된다.

### MCP와 샌드박스의 관계

`workspace-write` 샌드박스는 인터넷 전체를 차단하지만, MCP는 "이 특정 서버만 콘센트처럼 연결"하는 방식이다. 더 정밀하고 안전한 확장이다. `danger-full-access`가 인터넷 전부를 여는 것이라면, MCP는 필요한 구멍만 뚫는 것이다.
