# codex-05: 승인 + 샌드박스 3모드 — "AI에게 열쇠를 얼마나 줄까"

> **레슨 목표**: Codex의 샌드박스 3모드(read-only / workspace-write / danger-full-access)와 승인 정책의 의미를 이해하고, 실제로 모드를 바꿔가며 같은 요청의 행동 차이를 체감한다.
>
> **예상 시간**: 50분 (개념 도입 10분 + 실습 1 모드 체험 15분 + 실습 2 실제 데이터 분석 20분 + 실습 3(선택) 5분)
>
> **사전 준비**: 터미널 1개 + Codex 설치·로그인 완료(`codex --version` 확인) + `~/practice-data/creator/` 폴더 접근 가능 (경로가 다르면 실습 각 단계의 `cd` 명령에서 실제 경로로 바꿔주세요)
>
> **진입 커맨드**: `/codex-05`

---

🔎 **명령·모델·플래그 안내**: 아래 명령어·플래그·모델명(gpt-5.x)은 codex 0.140.0 기준(실제 0.140.0 실측)입니다.

---

## 📚 학습 목표 — 4가지 도달/와우 모먼트

1. **"열쇠를 내가 쥔다"** — Codex가 내 파일을 얼마나 다룰 수 있는지 내가 직접 결정한다는 것을 실감한다
2. **"보기만 해, 손대지 마"가 된다** — `read-only` 모드를 켜면 Codex가 파일을 아무리 분석해도 한 글자도 못 고치는 것을 직접 확인한다
3. **기본 모드(workspace-write)의 정체를 안다** — 기본값이 "묻지도 않고 다 해버리는 모드"가 아니라 "인터넷은 막고, 위험한 건 물어보는 안전 모드"임을 이해한다
4. **모드를 상황에 맞게 고른다** — "처음 탐색 때는 read-only, 실제 작업은 workspace-write" 패턴이 손에 익는다

## 🎯 개념 도입 (10분) — 새 비서에게 열쇠를 얼마나 줄까

### 비유: 신뢰가 쌓이는 만큼 열쇠를 준다

지우가 새 비서를 채용했다고 상상해 보세요.

첫날, 지우는 비서에게 사무실 열쇠를 어디까지 줄지 고민합니다.

**1단계 — 열람만**: "서류 봐도 돼, 하지만 만지지는 마."
비서는 지우의 파일을 읽고 현황을 파악할 수 있지만, 뭔가를 고치거나 지우는 건 금지. 서류 정리, 파일 수정, 이메일 발송 — 전부 불가. 처음 감을 잡는 첫날에 딱 맞는 수준.

**2단계 — 사무실 안 자율**: "사무실 안에서는 알아서 일해, 단 밖에 전화하거나 다른 데 접촉할 때는 먼저 물어봐."
비서는 지우의 파일을 읽고·정리하고·초안을 만들고·저장할 수 있어요. 하지만 인터넷으로 뭔가를 보내거나 외부에 연락하는 건 먼저 허락을 받아야 해요. 대부분의 일상 업무에 딱 맞는 수준. **이게 Codex의 기본값 = workspace-write 모드입니다.**

**3단계 — 완전 위임**: "모든 시스템 접근 권한 다 줄게, 인터넷도 마음대로 써."
비서는 인터넷도 쓰고, 외부 서비스도 연결하고, 하고 싶은 건 다 할 수 있어요. 충분히 신뢰가 쌓인 뒤, 정말 필요한 자동화 시나리오에서만 쓰는 수준.

Codex의 샌드박스(sandbox) 3모드가 정확히 이 3단계입니다.

> 💡 **샌드박스(sandbox)** = 모래놀이판. 아이가 모래놀이판 *안에서만* 놀게 하는 것처럼, Codex가 지정된 범위 *안에서만* 일하게 하는 안전 울타리. 모드를 올릴수록 울타리가 넓어집니다.

---

### 샌드박스 3모드 한눈에 비교

| 모드 | 파일 읽기 | 파일 수정·저장 | 명령 실행 | 인터넷 연결 | 언제 쓰나 |
|------|:---:|:---:|:---:|:---:|---|
| **read-only** | ✅ | ❌ | ❌ | ❌ | 처음 탐색·감 잡기 / "망칠까봐 무서울 때" |
| **workspace-write (기본값)** | ✅ | ✅ | ✅ | ❌ | 평소 일상 작업 대부분 |
| **danger-full-access** | ✅ | ✅ | ✅ | ✅ | CI 자동화 등 충분히 검증한 뒤에만 |

**핵심**: 기본값(옵션 없이 실행)이 `workspace-write`이며, 이미 인터넷은 차단된 안전한 상태입니다. "기본값이 위험한 모드 아닐까?" 걱정은 안 해도 됩니다.

> 💡 **"danger(위험)"라는 이름이 붙은 이유**
> 세 번째 모드 이름에 "danger"가 들어가 있습니다. 겁주려는 게 아니에요. "이 모드는 인터넷에도 접근하니 충분히 이해하고 써야 해요"라고 알려주는 경고 표시입니다. 마치 약통에 "전문가 상담 후 복용"이라고 적힌 것처럼요.

---

### 승인 정책 — "위험한 건 물어봐"의 세부 조절

샌드박스 모드가 "어디까지 들어갈 수 있나"라면, **승인 정책(approval policy)**은 "들어가기 전에 물어볼까, 말까"를 결정합니다.

| 승인 정책 | 설명 | 플래그 |
|---|---|---|
| **on-request** (필요시 물어보기) | 모델이 "이건 물어봐야겠다"고 판단할 때만 멈추고 확인 | `--ask-for-approval on-request` |
| **untrusted** (신뢰 외 명령은 승인) | 신뢰 목록 밖 명령 실행 전 하나씩 승인 | `--ask-for-approval untrusted` |

> 💡 **on-request의 의미**: "매번 물어보기"가 아닙니다. 읽기 작업 등 안전한 작업은 그냥 진행하고, 모델이 "여기서는 확인이 필요하다"고 판단할 때만 멈춥니다. 가장 보수적으로 쓰려면 `--ask-for-approval untrusted`(신뢰 외 명령 전부 승인)를 쓰세요.

처음엔 `--ask-for-approval on-request`를 강력 추천합니다. 예상치 못한 일이 일어나도 "잠깐요!"라고 멈출 수 있거든요.

---

### 추가 안전장치 — 자동으로 보호되는 것들

모드를 어떻게 설정해도 자동으로 보호되는 것이 있습니다.

- **`.git` / `.codex` / `.agents` 폴더** → 항상 읽기 전용 보호 (설정·Git 이력을 실수로 망가뜨릴 위험 없음)
- **네트워크** → workspace-write 모드에서는 기본적으로 인터넷 차단
- **OS 샌드박스** → Mac은 Seatbelt, Linux는 bubblewrap, Windows는 native/WSL 방식으로 운영체제가 자체 격리. "OS가 한 겹 더 막아준다"고만 알면 됩니다.

### Claude Code와 비교하면

CC(Claude Code)에도 "어디까지 허락할지" 결정하는 기능이 있습니다. 두 도구는 같은 원리를 씁니다. 이름표만 조금 다를 뿐이에요.

- CC: 승인 게이트 (허락할 작업 지정)
- Codex: 샌드박스 3모드 + 승인 정책 (모드로 더 또렷하게 구분)

"AI에게 권한을 단계적으로 준다"는 생각은 두 도구 모두 같습니다.

## 🔧 실습 1 — 3가지 모드 직접 체험하기 (15분)

지우가 Codex를 처음 실무에 쓰는 날입니다. 같은 폴더에 대해 모드만 바꿔가며 무엇이 달라지는지 직접 확인합니다.

### 1단계: 터미널을 열고 실습 폴더로 이동

```bash
cd ~/practice-data/creator
```

> 💡 **경로 확인 팁**: `pwd` 명령을 치면 지금 어느 폴더에 있는지 확인할 수 있습니다. 경로가 다르다면 실제 위치로 바꿔주세요.

**왜**: Codex가 일할 폴더를 지정해야 해요. 이 폴더에 `course_sales_monthly.csv`, `student_inquiries.csv` 등 지우의 실습 데이터가 있습니다.

**기대 결과**: 터미널 프롬프트가 `creator` 폴더를 가리키는 것을 확인 (`pwd` 명령으로 확인 가능).

---

### 2단계: read-only 모드로 Codex 실행

가장 안전한 모드입니다. Codex가 파일을 읽기는 하지만 **절대로 수정하거나 새로 만들지 않습니다.**

```text
codex --sandbox read-only --ask-for-approval on-request "이 폴더에 어떤 파일들이 있는지 둘러보고, 지우라는 1인 사업자가 어떤 사업을 하는지 요약해 줘"
```

**뜯어보면**:
- `--sandbox read-only` → 읽기만 허용, 파일 수정·저장 불가
- `--ask-for-approval on-request` → 모델이 필요하다고 판단할 때 확인 요청 (모든 작업 전에 매번 묻는 건 아님)
- 따옴표 안 → 지우의 지시문

**기대 결과 (예시)**:

```
폴더를 탐색하겠습니다...

발견한 파일:
- course_sales_monthly.csv (강의별 월별 매출 기록)
- student_inquiries.csv (수강생 문의 내역)
- newsletter_stats.csv (뉴스레터 성과 데이터)
- content_plan.md (콘텐츠 기획 노트)
- research_memo.md (리서치 메모)
- consulting_schedule.md (컨설팅 일정)

요약:
지우는 온라인 강의 제작·판매, 주간 뉴스레터 발행, 소규모 컨설팅을 운영하는
1인 사업자입니다. 강의 5개 운영, 뉴스레터 구독자 약 5,755명(41호 기준,
2026-03-23), 매출 데이터는 2025년 7월부터 2026년 3월까지 기록되어 있습니다.
```

**관찰 포인트**:
- Codex가 폴더를 읽고 파일 목록을 정확히 나열했나요? ✓
- 파일 수정·저장 없이 읽기만 했나요? ✓ (`ls` 로 파일 수 확인 — 새 파일이 생기지 않았어야 함)
- 사업 현황을 자연스럽게 요약했나요? ✓

> 💡 **안심 포인트**: read-only 모드에서는 Codex가 아무리 열심히 일해도 내 파일이 달라지지 않습니다. 완전히 안전한 상태에서 "Codex가 내 폴더를 어떻게 이해하는지" 구경할 수 있어요.

---

### 3단계: "파일을 수정하라"고 시키면 어떻게 될까

read-only 모드의 핵심 특성을 직접 확인합니다.

```text
course_sales_monthly.csv 파일을 열어서 2026년 3월 데이터 아래에 "분석 완료"라고 한 줄 추가해 줘
```

**기대 결과**:

```
죄송합니다, 현재 read-only 샌드박스 모드로 실행 중이라 파일을 수정할 수 없습니다.
파일을 읽고 분석하는 것은 가능하지만, 저장·수정·삭제 작업은 제한되어 있습니다.
파일 수정이 필요하다면 --sandbox workspace-write 모드로 재실행해 주세요.
```

**체크포인트**: Codex가 정중하게 거부했나요? 이게 read-only의 핵심입니다 — 지우가 실수로 "파일 다 삭제해줘"라고 말해도 read-only 모드에서는 아무것도 지워지지 않아요.

---

### 4단계: workspace-write 모드로 파일 생성 체험

이제 Codex에게 진짜 일을 시켜봅니다. 분석 결과를 새 파일로 저장하게 합니다.

```text
codex -m gpt-5.5 --sandbox workspace-write "course_sales_monthly.csv 파일을 읽고, 매출 상위 강의 3개를 골라 한 줄씩 요약해줘. 결과를 sales_summary.txt 파일로 저장해줘."
```

**명령어 뜯기**:
- `--sandbox workspace-write` → 내 폴더 안 파일 읽기·쓰기 허용, 인터넷은 차단
- 파일을 읽고 → 새 파일을 만들어 저장하는 작업

**기대 결과**:

```
course_sales_monthly.csv 파일을 읽는 중...

데이터 분석 완료.

매출 상위 3개 강의 (순매출 기준):
1. 챗GPT로 콘텐츠 10배 빠르게 — 2026-01 순매출 1,252만원
2. 클로드코드로 업무 자동화 입문 — 2026-03 순매출 1,242만원
3. 1인 지식사업 수익화 로드맵 — 2026-01 순매출 1,167만원

sales_summary.txt 파일 생성 완료.
```

**체크포인트**: `ls` 명령으로 `sales_summary.txt` 파일이 생겼나요? read-only 때는 생기지 않았던 파일이 workspace-write 모드에선 **실제로 생겼습니다.** 이게 두 모드의 결정적 차이입니다.

> 💡 **workspace-write 모드의 기본 동작 — "follow-through"**
> workspace-write 모드에서 Codex는 작업 폴더 안에서는 읽기·수정·삭제까지 대체로 알아서 진행하고, 작업 폴더 밖이나 네트워크처럼 영향 범위가 큰 일에서 확인을 요청합니다. 파일 삭제·덮어쓰기처럼 되돌리기 어려운 일도, 모델이 위험하다고 판단하면 확인을 요청할 수 있어요(항상 보장되는 건 아닙니다). 이것을 "follow-through(되도록 끝까지 처리하되, 위험 판단 시 확인)"라고 부릅니다.
>
> 새 파일 만들기 → 되돌릴 수 있어서 알아서 진행 ✅
> 기존 파일 삭제·덮어쓰기 → 되돌리기 어려운 일, 모델이 위험하다 판단하면 확인 요청이 뜰 수 있음 ⚠️

---

### 5단계: 승인 정책(on-request)으로 확인 흐름 보기

기존 파일을 수정하는 것처럼 영향이 큰 작업에서 Codex가 확인을 요청할 수 있습니다. 그 흐름을 직접 봅니다. (모델이 안전하다고 보면 확인 없이 진행할 수도 있어요.)

```text
codex --ask-for-approval on-request "student_inquiries.csv에서 '실습' 카테고리 문의만 뽑아서 같은 파일에 정리해줘."
```

Codex가 확인이 필요하다고 판단하면, 이렇게 물어봅니다:

```
student_inquiries.csv 파일을 수정하려고 합니다.
'실습' 카테고리 행만 남기고 나머지를 필터링합니다.
진행할까요? (y/n)
```

여기서 **`n`(아니오)을 눌러보세요.**

**기대 결과**: Codex가 "알겠습니다, 중단합니다."라고 하고 아무것도 바꾸지 않습니다.

**체크포인트**: n을 눌렀을 때 파일이 실제로 변경되지 않았나요? 이 안전장치가 있으니 workspace-write 모드도 무섭지 않습니다.

---

### 6단계: `/permissions`로 현재 상태 확인하기

Codex 세션 안에서 현재 샌드박스 모드와 승인 정책을 확인합니다.

```bash
codex "안녕"
```

그리고 대화 중에 입력:

```text
/permissions
```

**기대 결과 (예시)**:

```
현재 권한 설정:
- 샌드박스 모드: workspace-write
- 승인 정책: on-request (모델 판단 시 확인)
- 네트워크 접근: 차단됨
- 보호된 경로: .git, .codex, .agents
```

**체크포인트**: "네트워크 접근: 차단됨"이 보인다면 workspace-write 모드가 확실히 인터넷을 막고 있다는 뜻입니다. `/status`로도 비슷한 정보를 확인할 수 있습니다.

### 4단계: `@파일` 방식으로 두 파일 함께 분석

### 5단계: 분석 결과를 파일로 저장하기 (workspace-write + on-request)

지금까지 read-only로 살펴본 내용을 종합해서, 이번엔 workspace-write 모드로 보고서 초안을 파일에 저장합니다. `on-request`를 함께 줘서 Codex가 필요하다고 판단할 때 내용을 먼저 확인합니다.

```text
codex -m gpt-5.5 --sandbox workspace-write --ask-for-approval on-request "지금까지 본 3개 파일(course_sales_monthly.csv, newsletter_stats.csv, student_inquiries.csv)을 종합해서 간단한 인사이트 노트를 만들어줘.

형식:
## 이번 달 핵심 발견 3가지
(강의 매출·뉴스레터·수강생 문의에서 한 가지씩)

## 다음 달 추천 행동 2가지
(데이터 기반으로)

파일명: monthly_insight_draft.md 로 저장해줘."
```

### 6단계: 위험한 작업 요청 — Codex가 멈추는 순간

Codex가 위험하다고 판단하는 작업을 시키면 어떻게 반응하는지 봅니다.

```text
분석이 끝난 course_sales_monthly.csv 파일 삭제해 줘. 필요 없어.
```

**기대 결과 (Codex가 확인을 요청하는 경우)**:

```
작업 유형: 영구 삭제 (복구 불가)

```

**체크포인트**: 확인 요청이 떴다면 `n`을 눌러 취소하세요. on-request에서는 Codex가 위험하다고 판단할 때 확인 요청이 뜰 수 있고, 그때 `n`으로 멈추면 됩니다. (확인 없이 바로 진행됐다면 모델이 안전하다고 판단한 것 — 그래서 정말 중요한 파일은 미리 복사해두는 습관이 안전합니다.)

## 📝 실습 3 (선택 — 시간이 남으면) — `config.toml` 미리보기

매번 `--sandbox read-only`나 `--ask-for-approval on-request`를 타이핑하기 번거롭다면, **기본값을 파일로 저장**해 둘 수 있습니다.

> 비유: 자주 가는 카페에 "아메리카노, 샷 추가, 덜 뜨겁게"를 단골 주문으로 저장해 두는 것. 매번 말 안 해도 됩니다.

config.toml은 이런 모양입니다:

```toml
model = "gpt-5.5"
model_reasoning_effort = "medium"
sandbox_mode = "workspace-write"
```

- `model` → 기본으로 쓸 모델
- `model_reasoning_effort` → 추론 강도(low / medium / high / xhigh)
- `sandbox_mode` → 기본 샌드박스 모드(read-only / workspace-write / danger-full-access)

> 💡 **이 파일 직접 만들기는 다음 레슨(codex-06)에서**: config.toml과 AGENTS.md(Codex 규칙서)는 한 쌍입니다. 다음 레슨에서 둘을 함께 만드는 실습을 합니다. 지금은 "이런 파일이 있구나" 정도만 확인하고 넘어가세요.

### 세션 중 모델 바꿔보기 — 실시간 전환

config.toml 없이도, 이미 실행 중인 Codex 세션 안에서 모델을 바꿀 수 있습니다.

```text
codex "안녕"
```

세션이 열리면:

```text
/model
```

**기대 결과 (예시)**:

```
사용 가능한 모델:
  [*] gpt-5.5 (현재)
      gpt-5.4-mini (빠름·경량, 단순 작업용)

변경하려면 모델 이름 입력: _
```

여기서 `gpt-5.4-mini`를 입력해 전환해 보세요. 응답 속도·깊이가 달라지는 걸 체감할 수 있습니다. 이게 codex-03에서 배운 "effort·모델 조절"의 실습 3 버전입니다.

## 🔍 흔한 오류와 해결법

| 상황 | 원인 | 해결 방법 |
|------|------|-----------|
| "파일을 수정할 수 없습니다" 메시지가 뜸 | read-only 모드로 실행 중 | `--sandbox read-only` 빼고 다시 실행, 또는 `--sandbox workspace-write`로 재실행 |
| Codex가 자꾸 "해도 될까요?" 물어봐서 번거롭다 | `--ask-for-approval untrusted` 승인 정책 상태 | `--ask-for-approval on-request`로 바꾸거나, 완전히 맡기려면 `--yolo`(외부 격리(Docker 등)가 있는 환경에서만). 뭘 시키는지 확실할 때만 |
| "인터넷에 접근할 수 없습니다" 에러 | workspace-write 모드는 네트워크 차단 상태 | 외부 API·웹 연결이 필요하면 `--sandbox danger-full-access`(충분히 검증 후). 또는 외부 데이터를 파일로 미리 받아놓고 시작 |
| 파일을 만들었는데 어디에 생겼는지 모르겠다 | 작업 디렉터리 확인 필요 | `/status`로 작업 디렉터리 확인. `--cd` 옵션으로 지정한 폴더 안에 생성됨 |
| `y/n` 질문이 뜨지 않고 바로 진행됨 | 읽기 작업은 승인 없이 진행이 정상. on-request는 모델이 필요하다 판단할 때만 멈춰 묻기 때문에 안 뜰 수도 있음 | 읽기 작업은 승인 불필요. 더 자주 확인받고 싶으면 `--ask-for-approval untrusted`로 바꿔본다 |
| `/permissions` 명령이 안 먹힘 | Codex 세션 밖에서 입력함 | `codex "..."` 로 세션을 먼저 시작하고, 대화 중에 `/permissions` 입력 |
| config.toml을 만들었는데 반영 안 됨 | 파일 위치가 잘못됨 | `cat ~/.codex/config.toml`로 위치 확인. `~/.codex/` 폴더가 없으면 먼저 `mkdir -p ~/.codex` 실행 |
| 승인 화면에서 `y` 눌렀는데 아무것도 안 됨 | 영어 `y`가 아닌 한글 입력 상태였을 수 있음 | 입력 모드 확인 후 영문 소문자 `y` + Enter |
| `cd` 후 경로를 잊어버렸다 | 현재 위치 확인 필요 | `pwd` 명령으로 현재 폴더 확인. `ls`로 파일 목록 확인 |

## 💡 핵심 규칙/개념 정리

### 샌드박스 3모드 — 한 줄 기억법

```
read-only        =  "보기만, 절대 안 고침"          ← 처음 탐색할 때
workspace-write  =  "내 폴더 안에서 자유롭게"        ← 일상 작업 (기본값)
danger-full-access = "인터넷까지 열어줄게"           ← 검증된 자동화에만, 신중히
```

### 승인 정책

| 정책 | 행동 | 플래그 |
|---|---|---|
| on-request | 모델이 필요하다 판단할 때만 확인 | `--ask-for-approval on-request` |
| untrusted | 신뢰 목록 밖 명령 전부 승인 | `--ask-for-approval untrusted` |

> ⚠️ **`--yolo`(`--dangerously-bypass-approvals-and-sandbox`)는 승인 정책 값이 아닌 별도 우회 플래그**입니다. 승인과 샌드박스를 동시에 우회하므로, **외부 격리(Docker·CI 샌드박스 등)가 있는 환경에서만** 사용하세요. 로컬 개인 파일에 바로 적용하면 되돌릴 수 없는 변경이 생길 수 있습니다.

### 자동 보호되는 것 3가지

1. `.git` / `.codex` / `.agents` 폴더 → 항상 읽기 전용
2. 네트워크 → workspace-write 모드에서는 기본 차단
3. OS 샌드박스 → Mac Seatbelt / Linux bubblewrap이 한 겹 더 보호

### 실제 명령 패턴 정리

```bash
# 읽기만 (가장 안전)
codex -m gpt-5.5 --sandbox read-only "지시문"

# 기본 (일상 작업 — workspace-write가 기본값)
codex -m gpt-5.5 --sandbox workspace-write "지시문"

# 고치기 전에 필요시 물어보게 하기
codex -m gpt-5.5 --sandbox workspace-write --ask-for-approval on-request "지시문"

# 신뢰 외 명령은 하나씩 승인 (더 보수적)
codex -m gpt-5.5 --ask-for-approval untrusted "지시문"

# 전부 위임 (외부 격리 환경에서만 — 경고 숙지 후)
codex -m gpt-5.5 --yolo "지시문"

# 비대화형 자동 실행 (CI/CD 등)
codex exec --sandbox workspace-write "지시문"

# 세션 중 현재 상태 확인
/status
/permissions

# 세션 중 모델 전환
/model
```

### CC(Claude Code)와 비교

| 개념 | Claude Code | Codex |
|---|---|---|
| 안전 울타리 | 승인 게이트 | 샌드박스 3모드 |
| 기본 동작 | 위험한 건 확인 요청 | workspace-write 모드 (인터넷 차단, 위험한 건 확인) |
| 원리 | **동일** — "위험·영향 큰 작업은 확인을 거친다" | 동일 |

**이 레슨의 핵심 한 줄**: "기본값(workspace-write)은 이미 안전하게 설계돼 있고, read-only는 더 조심하고 싶을 때, danger-full-access는 정말 확실할 때 쓴다."
