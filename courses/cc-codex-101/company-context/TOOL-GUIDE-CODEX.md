## 1인 사업자 '지우'가 CC 옆에 두고 쓰는 Codex 치트시트

> **이 문서는 누구를 위한가**: 비개발자 1인 사업자. 터미널에 막 익숙해지는 중이고, 이미 Part 1에서 Claude Code(이하 CC)를 익혔다. 이제 **두 번째 운영 파트너인 Codex(OpenAI가 만든 AI 코딩 도구)**를 같은 컴퓨터에 들이려 한다.
>
> **목표**: Codex의 설치·로그인·핵심 명령·핵심 개념을 5분 안에 훑고, 실습 도중 "어? 이거 CC에선 `/`였는데 Codex에선 뭐였지?"라고 헷갈릴 때 CC와 나란히 펼쳐 보는 치트시트.
>
> **대원칙(딱 한 줄)**: **CC를 익혔으면 Codex는 90%가 그대로 옮겨온다.** 규칙서·샌드박스(안전 울타리)·MCP(외부 도구 연결)·effort(추론 강도)라는 뼈대는 똑같고, 회사·모델 이름·명령 기호 몇 개만 다르다. 이 문서는 그 "다른 10%"에 형광펜을 친다.
>
> ⚠️ **버전 안내**: 아래 명령·플래그·모델 이름은 codex-101 가이드(2026-06-18 스냅샷) 기준으로 정리했습니다. 모델 이름(gpt-5.x)이나 정확한 플래그는 빠르게 바뀔 수 있어, **실제로 강의·실습(촬영)을 진행하기 직전 공식 문서(`developers.openai.com/codex`)로 한 번 확인**하는 걸 권합니다. 이 문서에서 단정하기 애매한 부분은 **〔확인 필요〕**로 표시해 두었고, 그 항목은 실제 Codex 0.140.0 실측 단계에서 확정합니다.

## 0. 한 장 요약 — CC ↔ Codex 대응표

헷갈릴 때 여기부터 보세요. **왼쪽(CC, Part 1에서 배운 것)을 알면 오른쪽(Codex)은 거의 그대로입니다.**

| 무엇을 | Claude Code (Part 1) | Codex (Part 2) | 한 줄 |
|---|---|---|---|
| 만든 곳 | Anthropic (Claude 만든 곳) | OpenAI (ChatGPT 만든 곳) | 회사만 다름 |
| 설치 | `npm install -g @anthropic-ai/claude-code` | `npm install -g @openai/codex` | 둘 다 npm 한 줄 |
| 로그인 | `claude login` (Anthropic 계정) | `codex login` (ChatGPT 계정) | 계정만 다름 |
| 첫 실행 | `claude "..."` | `codex "..."` | 똑같이 따옴표 안에 질문 |
| 규칙서 | `CLAUDE.md` | **`AGENTS.md`** | 이름만 다른 같은 파일 |
| 스킬 부르는 기호 | `/스킬이름` | **`$스킬이름`** | 슬래시 → 달러 |
| 모델·생각 강도 바꾸기 | 모델·effort 선택 | `/model` | 같은 개념 |
| 안전장치 | 승인 게이트 | **샌드박스 3모드** + 승인 정책 | 모드 이름만 차이 |
| 외부 도구 연결 | MCP | MCP | **완전 동일** |
| 긴 대화 정리 | `/compact` | `/compact` | 동일 |

**한 문장 요약**: "CC와 Codex는 **같은 뼈대**(규칙서·샌드박스·MCP·effort)를 쓰는 형제 도구다. 슬래시(`/`)가 달러(`$`)로, `CLAUDE.md`가 `AGENTS.md`로 바뀐 정도다."

> 🎯 **forward-utility**: 이 대응표는 Part 2 전체(codex-01~11)의 뼈대이자, Part 4+ 멀티에이전트에서 "CC 에이전트와 Codex 에이전트를 한 팀에 섞어 쓸 때" 다시 펼쳐 보는 지도입니다.

## 0.5. 읽기 전에 — 불안 한 줄씩 미리 풀기

새 AI 도구를 깐다고 하면 보통 이런 걱정이 듭니다. 먼저 안심하고 시작하세요.

- *"또 새로 다 배워야 해?"* → 아니요. CC와 **90%가 같습니다.** 설치 → 로그인 → 폴더 지정 → 지시. 흐름이 똑같습니다.
- *"내 파일을 망가뜨리면?"* → 기본값이 **안전 모드**라, 되돌릴 수 없는 위험한 작업은 "이거 해도 될까요?"라고 먼저 묻습니다(F장).
- *"명령어를 다 외워야 하나?"* → 아니요. 이 치트시트를 켜놓고 그때그때 보면 됩니다. 외우는 건 시간이 알아서 해줍니다.
- *"개발자가 아닌데 가능해?"* → 가능합니다. 지우가 매출 정리·고객 문의 분류·강의 스크립트 다듬기를 하는 데 코딩 지식은 필요 없습니다.

## A. Codex가 한 줄로 뭔가

**Codex = 내 컴퓨터의 파일을 직접 읽고·고치고·실행하고·설명하는 AI 코딩 에이전트.** 만든 곳은 OpenAI(ChatGPT를 만든 회사).

CC를 "내 사업의 운영 파트너"로 들였던 것과 **정확히 같은 결**입니다. 차이를 굳이 비유하자면:

- **Claude Code** = 한 명의 든든한 비서. 긴 대화를 통째로 기억하며 한 호흡으로 깊게 파고든다.
- **Codex** = 여러 창구(터미널·앱·코드 편집기·웹·모바일)에서 같은 맥락으로 일을 이어받는 비서.

### 지우에게 왜 좋을까

지우의 사업 폴더에는 이런 파일들이 쌓여 있죠.

- 매출 정리 스프레드시트(CSV)
- 강의 슬라이드, 수강생 Q&A 메모
- 뉴스레터 초안, 이메일 답장 템플릿
- 고객 피드백 모음

지금까지는 이걸 **하나씩 열어서 → 복사 → AI 채팅창에 붙여넣기 → 답 복사 → 다시 파일에 붙여넣기**를 반복했을 겁니다. Codex는 이 왕복을 없앱니다. 폴더를 가리키고 "이거 분석해 줘"라고 말하면, **Codex가 알아서 파일을 찾아 읽고 작업**합니다.

그래서 이 코스는 "둘 중 뭐가 낫냐"를 가리지 않습니다. **둘 다 내 사업의 운영 파트너로 만드는 것**이 목표예요.

## B. 처음 10분 — 설치·로그인·첫 실행

CC 설치가 "한 번만 하는 일"이었던 것처럼, Codex도 처음 한 번만 깔면 됩니다.

### B1. 설치 (세 가지 길 중 택1)

```bash
# 길 1) npm 으로 (CC를 npm 으로 깔았으면 이게 제일 친숙)
npm install -g @openai/codex

# 길 2) Homebrew 로 (Mac 사용자)
brew install codex

# 길 3) 설치 스크립트로 (한 줄 복붙)
curl -fsSL https://chatgpt.com/codex/install.sh | sh
```

**무엇인가**: "Codex"라는 프로그램을 내 컴퓨터에 다운로드·설치하는 것. CC를 깔 때 `@anthropic-ai/claude-code`였다면, Codex는 `@openai/codex`입니다. 회사 이름만 바뀐 셈이에요.

**설치 확인**: 터미널에 `codex --version` 입력 → 버전 숫자가 뜨면 성공.

> 😟 **`command not found`가 떴다면?** 당황하지 마세요. 거의 항상 "터미널이 방금 깐 프로그램을 아직 못 찾는" 일시적 문제입니다. **터미널 창을 완전히 닫았다가 다시 열고** `codex --version`을 한 번 더 쳐보세요. 대부분 해결됩니다.

> 🪟 **Windows 사용자라면**: Codex는 WSL(윈도우 안의 리눅스 환경) 위에서 가장 잘 돌아갑니다. CC를 WSL에서 썼다면 같은 WSL 터미널에 그대로 위 명령을 입력하면 됩니다.

> 💡 Codex는 터미널(CLI)뿐 아니라 **데스크톱 앱(Mac/Windows)·코드 편집기 확장·웹/모바일**로도 쓸 수 있습니다. 하지만 우리 코스는 "나중에 AI 에이전트를 붙이려면 터미널이 필수"라서 **CLI(터미널)를 기본**으로 갑니다. 다른 창구는 부록(codex-04)에서 구경만 해요.

---

### B2. 로그인

```bash
codex login
```

**무엇인가**: Codex에게 "나 누구야"를 알려주는 과정. **ChatGPT 계정으로 로그인하는 게 권장 방법**입니다(평소 ChatGPT 쓸 때 그 계정). 브라우저가 열리고, 로그인·승인하면 끝.

**화면 흐름**: `codex login` 입력 → 브라우저 자동 열림 → ChatGPT 로그인 → "승인" 누르기 → 터미널로 돌아오면 준비 완료.

> 📌 **CC와 비교**: CC가 `claude login`(Anthropic 계정)이었다면, Codex는 `codex login`(ChatGPT 계정). **명령 모양도 거의 똑같습니다.**

> 💡 **대안(지금은 몰라도 됨)**: 회사·자동화 환경에서는 API 키(`export OPENAI_API_KEY=...`)로 로그인하기도 합니다. 지우처럼 혼자 쓰는 1인 사업자는 `codex login`(ChatGPT 계정) 한 줄이면 충분해요.

### B3. 첫 실행 (와우 모먼트)

내 사업 폴더로 들어가서 Codex를 깨워봅니다.

```bash
# 폴더 안의 파일을 대상으로 일하게 하려면 --cd 로 폴더를 지정
codex --cd ~/MyBusiness "이 폴더에 뭐가 있는지 둘러보고, 지난 3개월 어떤 일을 했는지 요약해 줄래?"
```

뜯어보면:
- `codex` → Codex 실행
- `--cd ~/MyBusiness` → "이 폴더 안의 파일들을 대상으로 일해" (CC의 작업 폴더 지정과 같은 역할)
- `"이 폴더에 뭐가 있는지…"` → 지우의 지시문(=프롬프트)

CC에서 `claude "..."`로 첫 대화를 시작했던 것과 **판박이**입니다.

**체험 포인트**: Codex는 답만 주는 게 아니라, "이 파일을 읽어볼게요 → 이건 매출 데이터네요 → 정리하면…" 식으로 **스스로 행동하며 일을 끝까지** 합니다. 위험하지 않고 되돌릴 수 있는 단계는 알아서 진행하고, 파일을 지우거나 덮어쓰는 것처럼 위험한 작업은 **"이거 해도 될까요?" 하고 먼저 물어봅니다**(이걸 follow-through, '끝까지 처리하되 위험할 땐 묻기'라고 부릅니다).

**🎯 Part 2 실습 codex-01에서**: 이 설치 → `codex login` → 첫 실행을 "맨바닥 새 컴퓨터" 가정으로 차근차근 따라 합니다. (Part 1 cc-01과 짝을 이루는 클립.)

## C. 핵심 명령어 — 터미널에서 입력하는 `codex ...`

매일 쓸 명령을 CC와 짝지어 놓았습니다. **CC에서 본 명령을 떠올리면 Codex 쪽은 자동으로 외워집니다.**

### 기본 틀

```bash
codex [옵션] "지시문"
```

### 외워둘 옵션 (이 표만 알면 80%)

| 옵션 | 예시 | 무엇에 쓰나 |
|---|---|---|
| `--cd` | `codex --cd ~/강의관리` | **어느 폴더에서 일할지** 지정 (거의 항상 씀) |
| `--add-dir` | `codex --cd ~/올해 --add-dir ~/작년` | 작업 폴더 말고 **다른 폴더도 함께** 읽히기 (예: 작년 자료가 딴 폴더에) |
| `-m` | `codex -m gpt-5.4` | 어느 모델을 쓸지 (E장) |
| `--ask-for-approval` | `codex --ask-for-approval on-request` | 파일 고치기 전에 "괜찮나요?" 묻게 하기 |
| `--sandbox` | `codex --sandbox read-only` | 안전 모드 골라 켜기 (F장) |
| `--full-auto` | `codex --full-auto` | 되돌릴 수 있는 작업은 묻지 않고 알아서 진행 |

### 채팅 안에서 `@파일` — 특정 파일 콕 집기

대화 도중, 특정 파일만 정확히 가리키고 싶으면 `@`를 씁니다.

```
@practice-data/sales_data.csv 이 파일에서 11월 대비 12월에 줄어든 채널만 알려줘
```

**무엇에 쓰나**: "그 파일 어딨더라" 헤매지 않고 정확히 지목. **CC도 `@파일` 문법을 그대로** 씁니다 — 표기가 동일하고, 파일을 정확히 지정할수록 답 품질이 좋아지는 것도 같습니다.

### 하루 워크플로우 (지우 버전)

```bash
# ① 일할 폴더 지정해서 시작
codex --cd ~/온라인강의

# ② 일 시키기 (자연스러운 한국어로)
"2026년 강의 매출을 월별로 정리해 줘"
"Part 1 강의 스크립트 글자 수 세어 줘"
"고객 리뷰에서 자주 나온 키워드 뽑아 줘"

# ③ 검토: Codex가 "이 파일 고칠게요. 괜찮나요?"라고 물으면 내용 보고 yes / no
```

**🎯 다시 쓰이는 곳**: 이 기본 호출은 codex-08(비대화형 자동화 `codex exec`)에서 "사람 없이 자동 실행"으로 확장됩니다. 배우는 곳 = codex-01(첫 실행)·codex-02(CLI 사용법).

## D. ✨ `$` 스킬 호출 — Codex만의 호출 기호 (가장 헷갈리는 차이)

여기가 **CC와 가장 또렷이 다른 지점**이라 따로 칸을 뺐습니다.

- **Claude Code**: 기능·스킬을 부를 때 보통 **`/`(슬래시)**로 시작합니다. (예: `/imagen`)
- **Codex**: 미리 만들어 둔 **스킬(반복 작업 묶음)을 부를 때 `$`(달러 기호)**를 씁니다.

| 하고 싶은 것 | Claude Code | Codex |
|---|---|---|
| 이미지 만들기 | `/imagen ...` | `$imagen ...` |
| 특정 스킬 부르기 | `/스킬이름` | `$스킬이름` |

**비유**: 슬래시(`/`)가 "메뉴판에서 고르기"라면, 달러(`$`)는 "단축번호로 내가 자주 쓰는 묶음 부르기"에 가깝습니다. 스킬 = "이 순서대로 일하라고 미리 적어둔 작업 묶음"이에요.

**지우의 활용 상상**: 매달 똑같이 "이번 달 매출 CSV 읽기 → 채널별 정리 → 전월 대비 증감 → 한 줄 코멘트"를 한다면, 이 4단계를 `$매월매출정리` 스킬로 묶어두고 매달 `$매월매출정리` 한 번이면 됩니다.

> ⚠️ **〔확인 필요〕**: `$`로 부르는 스킬의 정확한 등록 방법·이름 규칙은 Codex 버전에 따라 다를 수 있어, 실습 직전 실제 Codex 0.140.0 검증(실측 담당)·공식 문서로 확정합니다. 단, **모델·세션 관련 명령(`/model`·`/status`·`/compact` 등)은 Codex에서도 `/`를 그대로** 씁니다(아래 D장 참조) — `$`는 주로 "스킬 호출"에 쓰입니다. 이 문서에서는 "**CC의 `/`에 해당하는 Codex의 스킬 호출 기호 = `$`**"라는 핵심 차이만 확실히 기억하세요.

**🎯 Part 1 cc-08과 연결**: CC에서 배운 "스킬(SKILL.md)·슬래시 명령"의 개념이, Codex에선 `$`로 부르는 형태로 이어집니다. 개념은 같고 호출 기호만 달라요. 직접 비교 시연 = codex-02.

## E. 슬래시 명령 `/` — Codex와 대화하는 도중 쓰는 지시

Codex를 켜고 대화하는 **세션 안에서** 슬래시(`/`)로 시작하는 명령을 입력할 수 있습니다. CC의 슬래시 명령과 **똑같은 위치(대화 중)에서** 씁니다.

## E. 모델과 effort — "얼마나 똑똑하게, 얼마나 빠르게"

> 비유: **자동차 운전 모드**입니다. 급할 땐 빠른 모드, 중요한 길엔 신중 모드. 일에 맞춰 골라 씁니다.

CC에서 모델·effort를 골랐듯, Codex도 **모델(능력)**과 **effort(추론 강도)** 두 손잡이로 **속도·비용·품질의 균형**을 잡습니다. 개념은 100% 같고, 이름표만 다릅니다.

### E1. 모델 — 능력과 속도의 트레이드

⚠️ **모델명은 가장 빨리 바뀌는 부분입니다 — 시연 직전 `/model`로 실제 목록을 확인하세요.** (아래는 codex-101-정리.md 2026-06-18 스냅샷 기준 예시)

| 모델 (예시) | 성격 | 지우가 쓸 때 |
|---|---|---|
| `gpt-5.5` ⭐ | 가장 깊고 정확 (대신 느리고 비쌈) | 복잡한 멀티파일 작업, 중요한 콘텐츠·커리큘럼 기획 |
| `gpt-5.4` | 권장 기본값 (균형) | 평소 일상 작업 대부분 |
| `gpt-5.4-mini` | 빠르고 가벼움 | 단순 반복(주석 한글로, 짧은 변환, 이름 정렬) |
| `gpt-5.3-codex` | 옛 모델(은퇴 예정) | 쓰지 않음 |

```bash
# 시작할 때 모델 지정
codex -m gpt-5.5 "올해 강의 전체 커리큘럼을 다시 설계해 줘"

# 세션 도중 가볍게 전환
/model gpt-5.4-mini
```

### E2. effort — 추론 강도(신중함의 다이얼)

effort는 "Codex가 답하기 전에 **얼마나 깊이 생각할지**"입니다. 낮을수록 빠르고, 높을수록 신중·정확합니다.

```
none  →  low  →  medium  →  high  →  xhigh
(빠름·즉답)          (균형)         (신중·검증·길음)
```

- `none`/`low`: 짧은 변환·단순 실행. "이 문장 영어로 바꿔줘" 같은 일.
- `medium`: 여러 문서를 합치거나 조금 복잡한 작업. 평소 기본으로 무난.
- `high`/`xhigh`: 위험하거나 여러 파일을 건드리는 작업. 영향 범위까지 점검하고 검증 보고까지 해줌.

### E3. 🎬 같은 부탁, effort만 바꿔보기 (시연 핵심 — codex-03)

같은 부탁 `"이 매출 정리 함수에 오류 대비 처리를 넣어줘"`를 effort만 바꿔 던지면:

- **effort=low** → 바로 고친 코드 + 짧은 설명. (빠름)
- **effort=high** → 먼저 "이 함수를 어디서 쓰는지" 주변까지 점검 → 수정 → "이렇게 테스트해 보세요" 제안 → 검증 보고. (느리지만 꼼꼼)

**화면 대비 한 줄**: **effort ↑ = 더 신중·검증 / effort ↓ = 더 빠름.** 단순한 일엔 낮게, 위험하거나 복잡한 일엔 높게.

**쉬운 비유**: 급한 메모는 빠르게, 중요한 제안서는 꼼꼼하게 — 일에 따라 비서에게 들이는 공을 조절하는 것과 같습니다.

### E4. 설정 파일로 기본값 고정 (`config.toml`)

매번 손으로 고르지 않고, 기본값을 파일로 적어둘 수 있습니다.

> 비유: 자주 가는 식당의 **단골 주문**을 저장해두는 것. 매번 "덜 맵게, 양 많이"를 말 안 해도 됩니다.

```toml
# ~/.codex/config.toml  (또는 프로젝트 폴더의 .codex/config.toml)
model = "gpt-5.5"
model_reasoning_effort = "medium"
approval_policy = "on-request"
```

읽는 법:
- `model = "gpt-5.5"` → 기본 모델은 gpt-5.5
- `model_reasoning_effort = "medium"` → 기본 생각 강도는 중간
- `approval_policy = "on-request"` → 고치기 전에 매번 확인받기

이제 `codex --cd ~/강의관리`만 쳐도 이 설정이 자동 적용됩니다.

> 💡 **CC와 비교**: CC도 모델·생각 강도를 고릅니다. **"작업 난이도에 맞춰 능력·속도를 조절한다"는 원리는 두 도구가 똑같아요.** 이름(gpt-5.x vs Claude)과 설정 파일 형식만 다릅니다.

> ⚠️ **실습 주의(코스 불변 규칙)**: 모델·effort 전환은 **이론·미리보기 톤**으로 보여주거나 이미 배운 범위 안에서만. "아직 안 배운, 미리 세팅된 에이전트"로 건너뛰지 않습니다.

## F. AGENTS.md — Codex의 규칙서 (CC의 CLAUDE.md와 같은 것)

CC에서 만든 `CLAUDE.md`를 기억하시죠? "Codex가 내 프로젝트에서 지켜야 할 규칙"을 적는 파일이 **`AGENTS.md`**입니다. **이름만 다르고 역할은 똑같습니다.**

> 비유: **새 알바생에게 주는 업무 매뉴얼**입니다. 한 번 써두면 매번 같은 설명을 반복할 필요가 없습니다.

### F1. 무엇을 적나

- 내 사업·프로젝트가 뭔지(폴더 구조)
- 자주 쓰는 명령(빌드·테스트·정리 명령 같은 것)
- 지켜야 할 규칙(말투, 금지사항, 보고 형식)
- "완료"의 기준(무엇이 끝났다고 볼지)

### F2. 1인 사업자 지우의 AGENTS.md 예시

```markdown
# 내 콘텐츠 사업 규칙서 (AGENTS.md)

## 내 역할
나는 온라인 강의·뉴스레터를 운영하는 1인 사업자다. 코딩은 모른다.

## Codex의 역할
- 월간 매출 데이터 분석
- 고객 피드백 정리·분류
- 강의 스크립트 초안 검수

## 규칙
- 모든 보고는 비개발자가 읽기 쉬운 한국어로.
- 고객 실명·연락처는 절대 외부로 노출하지 말 것.
- 파일을 지우거나 덮어쓰기 전엔 반드시 먼저 물어볼 것.

## 완료 기준
- 분석 결과는 "그래서 다음에 뭘 하면 되는지" 한 줄을 꼭 포함한다.
```

### F3. 만들기 + 우선순위

```bash
codex /init    # 규칙서 뼈대를 자동으로 만들어 줌 (스캐폴드)
```

여러 위치에 규칙서가 있으면 **가까운 것이 이깁니다**(우선순위 높음 → 낮음):

```
AGENTS.override.md  >  ./AGENTS.md  >  ../AGENTS.md  >  ~/.codex/AGENTS.md
(프로젝트 강제 우선)   (현재 폴더)      (상위 폴더)      (내 컴퓨터 전체 기본)
```

쉽게 말해, **"이 프로젝트만의 규칙"이 "내 컴퓨터 전체 기본 규칙"보다 우선**합니다. 지금은 "프로젝트 폴더에 `AGENTS.md` 하나 두면 된다" 정도만 기억하세요.

> 💡 **CC와 비교**: `CLAUDE.md`(CC) ↔ `AGENTS.md`(Codex). **개념·작성법·우선순위 정신이 거의 같습니다.** Part 1에서 CLAUDE.md를 배웠다면, AGENTS.md는 "이름만 바꿔 끼우는 수준"으로 전이됩니다.

**🎯 forward-utility**: Part 1 `CLAUDE.md`(cc-05) → 여기 `AGENTS.md`(codex-06) → Part 3(지식관리/옵시디언)의 "내 노트 정리 규칙" → Part 4+(멀티에이전트)의 "여러 AI 팀원이 함께 읽는 규칙서". **규칙서 하나가 강의 전체를 관통하는 뼈대**입니다. codex-06에서 `AGENTS.md` + `config.toml`을 직접 만듭니다.

## G. 샌드박스 3모드 — "어디까지 허락할까" (안전장치)

**샌드박스(sandbox) = "모래놀이판"**. Codex가 내 컴퓨터를 함부로 헤집지 않도록 정해진 울타리 안에서만 놀게 하는 안전 공간입니다. 세 단계가 있고, **단계가 올라갈수록 Codex가 할 수 있는 일이 많아집니다.** CC의 승인 게이트와 같은 정신이며, Codex는 이를 3단계로 또렷하게 보여줍니다.

| 모드 | 읽기 | 쓰기(파일 수정) | 명령 실행 | 인터넷 | 한 줄 설명 |
|---|:---:|:---:|:---:|:---:|---|
| **read-only** | ✅ | ❌ | ❌ | ❌ | 보기만. 절대 안 고침. 가장 안전 |
| **auto (기본값)** | ✅ | ✅ | ✅ | ❌ | 내 폴더 안에서 읽고·고치고·실행. 인터넷은 차단 |
| **danger-full-access** | ✅ | ✅ | ✅ | ✅ | 인터넷까지 전부 허용. 위험 — 신중히 |

```bash
# 보기만 하게 (제일 안전, 처음엔 이걸 권장)
codex --sandbox read-only --ask-for-approval on-request "이 폴더 둘러보고 요약만 해줘"

# 내 폴더 안에서 알아서 일하게 (기본 자동 모드)
codex --full-auto "고객 피드백 정리해서 새 파일로 저장해 줘"

# 전부 허용 (CI 자동화 등, 충분히 검증한 뒤에만)
codex --dangerously-bypass-approvals-and-sandbox "..."
```

**쉬운 비유**: 새 비서에게 처음엔 "서류 보기만 해(read-only)" → 익숙해지면 "정리도 해(auto)" → 완전히 믿으면 "알아서 다 해(full-access)". **신뢰가 쌓이는 만큼 권한을 올리는 것**입니다.

**지우를 위한 안전 순서**:
1. 처음 만져볼 땐 **read-only**로 "보기만" 시키며 감을 잡으세요.
2. 익숙해지면 **auto(기본)**로 — 내 사업 폴더 안에서 읽고·고치고·정리하되 인터넷은 막힌, 딱 적당한 상태입니다.
3. **danger-full-access**(인터넷까지 전면 허용)는 진짜 필요할 때만, 무슨 일을 시키는지 확실히 알 때만. (거의 쓸 일 없습니다.)

**추가 안전장치(자동 적용)**: `.git`·`.codex`·`.agents` 같은 중요한 설정 폴더는 보호(읽기 전용)되고, 기본 작업 모드에서는 **인터넷이 차단**됩니다. 운영체제 차원의 울타리(Mac은 Seatbelt, 리눅스는 bubblewrap, Windows는 native/WSL)도 함께 작동합니다 — 그냥 "OS가 알아서 안전하게 막아준다"고만 알면 됩니다.

> 💡 기본 동작(follow-through): 되돌릴 수 있는 안전한 일은 알아서 진행하고, **되돌릴 수 없는 위험한 일**(파일 삭제 등)은 멈추고 "해도 될까요?" 하고 물어봅니다.

> 💡 **CC와 비교**: CC도 "어디까지 허락할지" 승인 게이트가 있습니다. **"읽기만 / 내 폴더 안에서 / 전부 허용"의 3단 사고방식은 두 도구가 같아요.** 모드 이름과 표기만 다릅니다.

**🎯 Part 2 codex-05에서**: 이 3모드를 직접 바꿔가며 "같은 부탁을 read-only로 시키면 안 고치고, auto로 시키면 파일을 만든다"를 체감합니다. "AI에게 권한을 단계적으로 준다"는 감각은 Part 4+ 멀티에이전트에서 여러 AI를 동시에 굴릴 때 안전의 핵심이 됩니다.

## H. MCP — 외부 도구 연결 (CC와 같은 표준)

MCP(Model Context Protocol, '모델-도구 연결 표준')는 **Codex에게 바깥 도구를 물려주는 콘센트**입니다. CC에서 배운 그 MCP와 **같은 표준**이에요 — 두 도구가 공유하는 거의 유일한 "완전 동일" 영역.

```bash
# 예시 1) OpenAI 개발자 문서를 Codex가 직접 찾아보게 연결
codex mcp add openaiDeveloperDocs --url https://developers.openai.com/mcp

# 예시 2) 웹 브라우저 조작 도구(Playwright) 연결
codex mcp add playwright -- npx --yes @playwright/mcp@latest

# 연결된 도구 목록 확인
codex mcp list
```

연결한 도구는 `config.toml`의 `[mcp_servers.*]` 항목으로 영구 저장됩니다.

> 💡 1인 사업자에게 MCP는 "처음부터 다 깔기"가 아니라 **필요해질 때 하나씩** 붙이는 영역입니다. Part 2 본편에서는 "이런 게 있다" 정도로 보고, 실제 연결은 codex-07(부록)에서.

> 💡 **CC와 비교**: **MCP는 두 도구가 공유하는 공통 표준**입니다. 연결 명령의 세부 모양만 다르고("`codex mcp add`" vs CC의 MCP 추가 방식), "외부 도구를 콘센트처럼 꽂는다"는 개념은 똑같아요.

**🎯 forward-utility**: MCP는 Part 3(지식관리)에서 **옵시디언/vault를 AI에 연결**할 때, Part 4+에서 여러 에이전트가 같은 외부 도구를 공유할 때 핵심이 됩니다. 배우는 곳 = codex-07 / Part 1 cc-09.

## I. 🌉 CC 안에서 Codex 같이 쓰기 — codex plugin

지우에게 가장 실용적인 부분이자, 두 도구를 잇는 **가장 중요한 다리**입니다. **"두 도구를 왔다 갔다 하기 번거로운데, 한쪽에서 같이 못 쓰나?"** → 됩니다.

- **codex plugin** = Claude Code 안에서 **Codex를 함께 호출**해 쓰는 공식 연계 방법입니다.
- 비유하자면, 평소 쓰던 CC라는 작업대 위에 **Codex라는 보조 도구를 끼워 넣는** 것. 큰 작업은 CC로 하다가, 특정 부분을 Codex에게 맡기고 싶을 때 한 작업대 안에서 넘길 수 있습니다.

**왜 좋은가(지우 관점)**: CC로 강의 기획을 진행하다가, "이 부분은 Codex(다른 엔진)의 시각으로도 한 번 봐줘" 하고 같은 자리에서 부탁할 수 있습니다. 창을 옮기지 않고 두 AI 파트너를 한 흐름에서 짝꿍처럼 부리는 거예요.

> ⚠️ **〔확인 필요〕**: codex plugin의 정확한 설치 경로·명령·세부 기능은 **실제 0.140.0 실측 + 공식 문서**로 확인한 뒤 실습 문서(cc-08 모듈)에 확정해 넣을 예정입니다. 이 치트시트에서는 "**CC 안에서 Codex를 공식적으로 같이 쓰는 길(codex plugin)이 있다**"는 큰 그림만 잡아두세요.

**🎯 forward-utility 1순위**: codex plugin은 **Part 1 CC 트랙(cc-08)에서 배우고 → Part 2 Codex 트랙으로 직결**되는 다리입니다. Part 4+ 멀티에이전트에서 "CC 에이전트 + Codex 에이전트 혼합 팀"의 출발점이 됩니다.

## L. 마지막 조언 — "CC를 아는 당신이 유리하다"

Part 1에서 CC를 익혔다면, Codex는 **새 도구가 아니라 같은 도구의 다른 브랜드**입니다. 새로 외울 건 사실상 몇 개뿐:

1. 규칙서 이름이 `AGENTS.md`라는 것
2. 스킬은 `$`로 부른다는 것
3. 샌드박스가 3모드(read-only / auto / danger)로 또렷하다는 것
4. 계정이 ChatGPT라는 것

처음 Codex를 만지면 "CC랑 비슷한데 미묘하게 다르네"라는 느낌이 들 겁니다. 그게 정상이고 사실 **좋은 신호**예요 — 뼈대가 같으니 헷갈리는 거고, 그 말은 곧 **하나를 익히면 나머지는 금방 따라온다**는 뜻이니까요.

지우의 목표는 "두 도구의 명령어를 다 외우는 것"이 아닙니다. **두 AI 파트너를 내 사업의 운영 팀으로 만들어, 혼자서도 마케팅·고객관리·매출분석·콘텐츠·리서치를 같이 끝내는 것**입니다.

명령어가 헷갈리면 언제든 이 문서로 돌아오세요. 자, 이제 Part 2 codex-01(Codex 설치와 첫 실행)부터 시작해 봅시다.

## 빠른 참조 (한 장 요약 — 잘라서 옆에 두세요)

**기본 명령**
```bash
codex --cd ~/프로젝트 "지시문"
```

**세션 중 자주 쓰는 / 명령**
```
/model        ← 모델·생각 강도 바꾸기
/permissions  ← 내 권한 확인
/plan         ← 계획 먼저 보기
/status       ← 사용량 확인
/compact      ← 긴 대화 요약 (작업 끝난 뒤에만)
```

**스킬 호출**
```
$스킬이름      ← Codex 스킬 부르기 (CC는 /스킬이름)
```

**안전 모드**
```bash
--sandbox read-only   ← 읽기만 (가장 안전)
(옵션 없음 = auto)    ← 기본, 위험한 건 물어봄
--dangerously-...     ← 완전 신뢰 (거의 안 씀)
```

**모델 고르기**
```bash
-m gpt-5.5       ← 깊고 정확 (느림)
-m gpt-5.4       ← 기본 (권장)
-m gpt-5.4-mini  ← 빠름 (단순한 일)
```
