## 핵심 가치와 일하는 방식

### 5가지 핵심 가치

1. **투명성 (Transparency)**
   - 진행 상황을 숨기지 않습니다. 막히는 부분도 조기에 공유합니다.
   - 모든 결정 근거를 팀과 함께 토론합니다.

2. **소유권 (Ownership)**
   - 각자 맡은 업무에 책임감을 갖습니다.
   - "누가 할 거야?"라는 질문보다 "내가 해야 한다"는 마인드셋.

3. **협력 (Collaboration)**
   - 개인의 이익보다 팀 전체의 성공을 우선합니다.
   - 동료의 성장을 돕는 것도 나의 책임입니다.

4. **품질 (Quality)**
   - 빨리 만들되, 제대로 만듭니다.
   - 기술 부채는 명확히 기록하고 계획적으로 갚습니다.

5. **호기심 (Curiosity)**
   - 새로운 기술과 방법론을 배우고 시도합니다.
   - 실패도 학습 기회로 봅니다.

### 일하는 방식의 원칙

| 원칙 | 설명 | 예시 |
|------|------|------|
| **비동기 우선 (Async First)** | 실시간 회의보다 문서와 비동기 피드백 선호 | 긴급 아님 → 메시지로 충분 |
| **문서화 의존 (Documentation Driven)** | 모든 중요 결정, 프로세스, 배운 점을 기록 | 회의 후 30분 내 요약 공유 |
| **작은 배포 (Ship Small)** | 한 번에 크게가 아닌 매주 배포 | 2주 스프린트, 주말 배포 |
| **피드백 문화 (Feedback First)** | 정기적인 1:1, 팀 회고, 사용자 피드백 | 주 1회 1:1, 월 1회 팀 회고 |

## 역할별 책임

### 팀장 (Product Lead)

**책임:**
- 팀의 전략 방향 수립 및 소통
- 분기별 목표 설정 및 진행 상황 모니터링
- 팀원 개발 계획 수립 및 성과 평가
- 인사 관리 (채용, 승진, 복지)
- 외부(경영진, 고객) 와의 이해관계 조율

**핵심 산출물:**
- 월간 팀 회의 (진행상황, 예정 업무)
- 분기 목표서 (OKR)
- 개인 성과 평가 (연 2회)

---

### PM (Product Manager)

**책임:**
- 고객 요구사항 수집 및 분석
- 제품 로드맵 작성 (3개월 선행)
- 기능 우선순위 결정
- 출시 전 제품 검증 (베타 테스트, 피드백 수집)
- 성공 지표(KPI) 정의 및 추적

**핵심 산출물:**
- 주간 우선순위 업데이트
- 기능 스펙시트 (개발팀과 협의)
- 월간 사용자 피드백 보고서

**일일 협업:**
- **개발팀**: 일일 스탠드업 (10분, 오전 9:30)
- **디자이너**: 기능 설계 리뷰 (주 2회)
- **QA**: 배포 전 검증 (배포 당일)

---

### 개발 리드 (Tech Lead)

**책임:**
- 기술 스택 선택 및 아키텍처 설계
- 코드 품질 기준 설정 및 리뷰
- 개발자 온보딩 및 기술 멘토링
- 성능 최적화 및 인프라 관리
- 기술 부채 추적 및 상환 계획

**핵심 산출물:**
- 기술 설계 문서 (큰 기능 전)
- 코드 리뷰 피드백 (매 PR)
- 월간 성능 리포트
- 기술 부채 관리표

**코드 리뷰 기준:**
- PR 제출 후 24시간 내 리뷰 시작
- 최소 1명 승인 후 머지 (긴급 수정 제외)
- 커밋 메시지: `[Feature|Fix|Refactor] 설명 (#이슈번호)`

### 개발자 (Developer)

**책임:**
- 할당된 기능/버그 구현
- 자신의 코드에 대한 테스트 작성 (단위 테스트 최소 70% 커버리지)
- 코드 리뷰 및 피드백 제공
- 성능 모니터링 및 개선
- 팀 문서 유지보수

**기대사항:**
- 일일 스탠드업 참여 (진행 상황, 막힌 부분 공유)
- 주간 코드 리뷰 최소 2건
- 월 1회 기술 공유 (새로 배운 것, 해결한 문제)

**기술 스택 (2026년 기준):**
- **프론트엔드**: React 18, TypeScript, Tailwind CSS
- **백엔드**: Python FastAPI, PostgreSQL
- **DevOps**: Docker, Kubernetes, GitHub Actions
- **모니터링**: DataDog, Sentry

### QA/테스터 (QA Engineer)

**책임:**
- 기능 테스트 계획 수립 (기능별 테스트 케이스 10~20개)
- 수동 테스트 실행 및 버그 보고
- 자동화 테스트 구성 및 관리
- 성능/보안 테스트 (월 1회)
- 배포 전 최종 검증

**테스트 정책:**
- 신규 기능 배포 전 필수 테스트: 기능 정상 동작, 엣지 케이스, 이전 기능 무회귀
- 버그 보고 양식: 제목, 재현 단계, 예상 동작, 실제 동작, 스크린샷/로그
- 회귀 테스트: 매 배포마다 (자동화 테스트 90%, 수동 10%)

```

├─ 월: 스프린트 계획 미팅 (10:00, 1시간)
│   └─ PM이 우선순위 기능 제시
│   └─ 팀이 스토리 포인트 추정 (1~13)
│   └─ 개발 리드가 기술 검토
│
├─ 화~목: 개발 & 코드 리뷰
│   └─ 일일 스탠드업 (9:30, 10분)
│   └─ 막힌 부분 즉시 공유
│
└─ 금: 스프린트 리뷰 & 회고 (14:00~15:00, 1시간)
    └─ 완료한 기능 데모
    └─ 팀 회고 (Good/Bad/Try)
    └─ 다음 스프린트 포커스 정의

주 2 (월~목)
├─ 개발 & 코드 리뷰 계속
└─ 목: 배포 준비 & QA 최종 검증
```

**스토리 포인트 기준:**
- **1**: 설정 변경, 문서 업데이트
- **3**: 단순 기능 (API 1개, 화면 1개)
- **5**: 중간 기능 (API 2~3개, DB 변경, 화면 2~3개)
- **8**: 복잡한 기능 (여러 API, 복잡한 로직)
- **13**: 매우 큰 기능 (분할 권고)

**월 배포 일정:**
- 월: 스프린트 1 완료 + QA 테스트
- 월: 스프린트 2 완료 + QA 테스트
- **주말 (토요일 오전 2시)**: 프로덕션 배포

배포 후 **월요일 09:00~10:00** 배포 후 모니터링 (개발 리드 + DevOps)

### 2. 버그/이슈 처리 (버그 바운스)

**버그 발견 → 보고 → 수정 → 배포 전 시간:**

| 심각도 | 설명 | SLA | 배포 시기 |
|--------|------|-----|----------|
| **P1 (Critical)** | 서비스 중단, 데이터 손실 | 1시간 내 인식 | 당일 핫픽스 배포 |
| **P2 (High)** | 주요 기능 오작동 | 2시간 내 | 다음 일일 배포 |
| **P3 (Medium)** | 부분 기능 오류, UX 개선 | 24시간 내 | 다음 스프린트 |
| **P4 (Low)** | 마이너 버그, 문서 오류 | 1주일 내 | 다음 월간 배포 |

**버그 보고 양식:**
```
제목: [P2] 로그인 화면에서 비밀번호 입력 후 엔터 안 먹힘

재현 단계:
1. 로그인 페이지 방문
2. 이메일 입력
3. 비밀번호 입력
4. 엔터 키 누름

예상 동작:
- 로그인 시도 (로딩 스피너 표시)

실제 동작:
- 아무것도 안 됨 (마우스 클릭하면 됨)

환경:
- 브라우저: Chrome 119, Safari 17
- OS: macOS 14.1

첨부:
- 스크린샷 / 비디오
```

### 3. 코드 리뷰 프로세스

**PR (Pull Request) 작성 -> 리뷰 -> 머지까지:**

```
1. 브랜치 생성
   git checkout -b feature/user-login-form

2. 커밋 (메시지 명확)
   git commit -m "[Feature] 사용자 로그인 폼 추가 (#1234)"

3. PR 제출
   - 제목: [Feature] 사용자 로그인 폼 추가
   - 설명:
     - 무엇을 했나: 새 로그인 폼 컴포넌트 추가
     - 왜 했나: 기존 폼 UX 개선
     - 테스트: 단위 테스트 12개 추가 (커버리지 85%)
     - 스크린샷: [before/after 이미지]
     - 체크리스트:
       ☑ 단위 테스트 통과
       ☑ 코드 스타일 준수
       ☑ 문서 업데이트 완료

4. 리뷰 (개발 리드 + 1명 이상)
   - 코드 품질: 가독성, 복잡도, 성능
   - 테스트: 커버리지, 엣지 케이스
   - 아키텍처: 설계 원칙 준수
   - 의견 예시:
     ✓ "코드 깔끔합니다"
     ? "이 함수는 왜 별도 분리했나요?"
     ✗ "변수명 'temp'가 아닌 'filteredUsers'로 부탁합니다"

5. 리뷰 반영 (작성자)
   - 모든 피드백에 답글 작성
   - 필요시 코드 수정 후 재커밋
   - 코드 리뷰 완료 후 대화: "반영했습니다"

6. 머지
   - 최소 1명 승인 후 "Squash and merge"
   - 스프린트 진행 중 즉시 배포 안 함 (스프린트 끝에 함께)
```

**코드 리뷰 기준 (팀 체크리스트):**
- [ ] 기능이 스펙을 만족하는가?
- [ ] 테스트 커버리지가 70% 이상인가?
- [ ] 성능이 조건(로딩 시간 3초 이내)을 만족하는가?
- [ ] 보안 (SQL injection, XSS, 인증 체크)은 안전한가?
- [ ] 기존 기능에 영향이 없는가?
- [ ] 코드 스타일이 일관적인가?
- [ ] 에러 처리와 로깅이 충분한가?

### 4. 배포 프로세스

**배포 전 체크리스트 (배포 담당: DevOps + Tech Lead):**

```
배포 전날 (금요일)
└─ QA 최종 검증 완료
   ├─ 새 기능 테스트
   ├─ 회귀 테스트 (주요 기능)
   └─ 성능 테스트 (로딩 시간, 메모리)

배포일 (토요일 01:30)
└─ 배포 전 (01:30~02:00)
   ├─ 데이터베이스 백업
   ├─ 현재 버전 스냅샷 저장
   ├─ 롤백 계획 검토
   └─ 모니터링 도구 활성화

배포 (02:00~02:30)
├─ 새 버전 릴리스 노트 발행
├─ 애플리케이션 배포
├─ 헬스 체크 (응답 속도, 에러율)
└─ 슬랙 #배포 채널에 완료 공지

배포 후 모니터링 (02:30~05:00)
├─ Datadog 대시보드 모니터링
├─ 사용자 에러 리포트 수집
└─ 필요시 즉시 롤백 준비
```

**롤백 (긴급 상황):**
- 심각한 버그 발견 → 즉시 이전 버전으로 복구 (5분 이내)
- 롤백 후 원인 분석 회의 (월요일 10:00)
- Post-mortem 문서 작성 (48시간 내)

| **Sentry** | 애플리케이션 에러 추적 | 24시간 | 프로덕션 에러 자동 수집 |

### 계정 설정 및 접근 권한

**온보딩 시 신청:**
1. **GitHub 계정**: 팀장에게 GitHub ID 제출 → 리뷰 권한 부여 (3일)
2. **Jira 계정**: IT 헬프데스크 신청 → 스프린트 보드 접근 (1일)
3. **Slack**: 회사 도메인 이메일로 자동 생성
4. **DataDog/Sentry**: 개발 리드가 추가 (첫 주)

**권한 기준:**
- 개발자: 모든 저장소 읽기, 주 저장소 쓰기
- QA: 모든 저장소 읽기, 이슈 생성/수정
- PM: GitHub 이슈/PR 읽기, Jira 전체 접근

## 자주 묻는 질문 (FAQ)

### 일정 & 근무

**Q: 재택근무 가능한가?**
A: 가능합니다. 월 1회 이상 오피스 방문 권장 (팀 미팅, 대면 협업). 월요일/수요일은 오피스 근무 강권합니다.

**Q: 야근은 얼마나 해야 하나?**
A: 정상적인 스프린트는 주 40시간입니다. 배포 주는 야근 가능성이 있습니다 (보상 휴가로 처리). 장시간 야근이 반복되면 일정 재조정이 필요합니다 (팀장 상담).

**Q: 휴가 신청은?**
A: Slack에서 `/request-time-off` 명령 사용 (팀장 자동 승인). 스프린트 중간은 피해주세요 (계획 수립 때 미리 공지).

**Q: 출장이 필요하면?**
A: 팀장에게 사유·일정·예산 함께 신청. 1주일 전 공지 권장. 원격 회의로 진행 가능하면 출장 취소.

---

### 업무 & 개발

**Q: 기능이 너무 복잡해 보이는데, 분할해도 되나?**
A: 권장합니다. 스토리 포인트 8 이상이면 분할 검토. PM과 함께 작은 단위로 나누는 것이 배포 속도와 품질 개선에 도움됩니다.

**Q: 버그를 발견했는데, 바로 고치면 안 되나?**
A: 심각도에 따라 다릅니다 (위 "버그/이슈 처리" 참조). P1은 즉시 핫픽스, P3/P4는 다음 스프린트 예약. 개발 중 발견한 버그는 현재 스프린트에서 처리.

**Q: 코드 리뷰에서 거절당했다. 뭐가 문제인가?**
A: 일반적인 거절 사유:
- 테스트 부족 → 테스트 추가
- 변수명 불명확 → 이름 변경
- 성능 우려 → 최적화 후 벤치마크 제시
- 아키텍처 위반 → 설계 문서 재검토

모든 거절에는 구체적 이유가 있습니다. 이해 안 되면 댓글로 질문하세요.

**Q: 야, 새로운 라이브러리 써도 되나?**
A: 개발 리드 사전 승인 필수. 이유: 보안, 유지보수성, 팀 표준화. 작은 라이브러리는 빠르게 승인됩니다 (24시간). 큰 통합(DB 전환 등)은 기술 리뷰 회의 필요.

### 성과 & 성장

**Q: 성과 평가는 어떻게 되나?**
A: 연 2회 (6월, 12월):
- **자기 평가**: 스프린트 기여도, 배운 점, 목표 달성도
- **동료 피드백**: 팀원 3~4명이 협력·코드 품질·커뮤니케이션 평가
- **팀장 평가**: 전체를 종합해 최종 등급 결정 (優秀/良好/合格)
- **급여/보너스**: 평가 결과, 회사 실적, 시장 급여 기준 고려

**Q: 기술 성장을 위해 뭘 할 수 있나?**
A: 여러 경로가 있습니다:
- 월간 기술 살롱 발표 (새 기술 경험담)
- 분기 1회 기술 도서/강의 지원 (예산 내)
- 오픈소스 기여 (업무 시간 일부 할당 가능)
- 팀 내 멘토링 (시니어 개발자)

팀장과 1:1에서 성장 계획을 수립하세요.

**Q: 캐리어는 어떻게 발전하나?**
A: 3가지 경로:
- **기술 심화** (시니어 개발자 → 아키텍트)
- **리더십** (개발 리드 → 팀장)
- **전문화** (풀스택 → 특정 영역 전문가)

팀장과 연간 1~2회 경력 상담을 합니다.

### 문제 해결 & 분쟁

**Q: 팀원과 의견이 안 맞을 때?**
A: 단계별 처리:
1. 당사자끼리 대면 토론 (Slack 아님, 실시간 대화)
2. 해결 안 되면 팀장 중재
3. 팀 전체 방침이면 회의에서 토론

대부분은 1단계에서 해결됩니다. 감정적으로 대하지 말고 논리적으로.

**Q: 실수를 한 건데, 보고하면 혼나나?**
A: 아닙니다. 우리 문화는:
- 실수 즉시 보고 (숨기는 것이 더 문제)
- 근본 원인 분석 (누구 탓 ❌)
- 개선 계획 수립 (재발 방지)

같은 실수 3회면 정책 재논의합니다.

**Q: 번아웃됐는데?**
A: 팀장에게 상담하세요. 가능한 조치:
- 업무 조정 (덜 중요한 업무 연기)
- 휴가 권장 (1주 권장)
- 역할 변경 (다른 팀원과 교차)

회사는 번아웃을 심각하게 봅니다. 숨기지 마세요.
